# Test suite quality audit — tests/ (excl. test_meta) and _benchmarks/

## Summary

Covered `tests/conftest.py` in full, 20+ full test files spanning `tests/` root, `tests/performance/kernel_tuning/`,
`tests/stats/`, `tests/system/`, plus both `_benchmarks/*.py` scripts, and structurally sampled (grep + targeted
reads) the remaining ~50 files. Every claim below was verified by reading the cited source and/or test lines, not
inferred from names. The suite has real strengths (the kernel-tuning-cache concurrency tests and the normality-test
statistical calibration are genuinely good engineering) but also a systemic pattern — concentrated in a cluster of
files that read as auto-generated during a "Phase 2 refactoring" / coverage-chasing pass (`test_system*.py`,
`test_distributed.py`, `test_web.py`, `test_numbalib.py`, `test_llm_providers.py`) — of assertions that cannot
actually fail no matter what the production code does, one file of tests that is permanently and silently skipped
because it targets a class that doesn't exist, one benchmark file that pytest never even collects, and undocumented/
inconsistent test-file organization (flat root vs. subfolders, "_extra"/"_extra2"/"_extra3" naming) that isn't
explained anywhere in TESTING.md.

Findings: 3 High, 10 Medium, 4 Low (listed below, most severe first; counted after, not before).

## Findings

### [High] Tautological assertion hides a fully-deterministic, verified-wrong-by-me-checking-source outcome — tests/test_serialization.py:198-207
- **Category**: correctness / assertion depth
- **Problem**: `test_unserialize_uncompressed_data` builds a raw (non-zlib) pickle blob and asserts `result is None or isinstance(result, dict)`. I traced `unserialize()` in `src/pyutilz/core/serialization.py:130-142`: `zlib.decompress` raises `zlib.error` on non-zlib input, is caught, falls back to `data = obj` (line 136-139), and `pickle.loads(data)` succeeds, so the **only possible correct output is `{"key": "value"}`** — the behaviour is fully deterministic, not "either/or" as the test's comment claims.
- **Failure scenario**: If a future edit narrows the `except zlib.error:` fallback (e.g. to `except zlib.error as e: raise` at that inner site, letting the outer blanket `except Exception` at line 145 catch it and log+return `None`), the function silently starts always returning `None` for every non-zlib-compressed payload ever passed to it — a real behavioural regression (data loss for any caller passing raw pickle bytes) — and this test still passes, because `None` satisfies the first half of the `or`.
- **Suggested fix**: `assert result == {"key": "value"}`.

### [High] Whole test file silently and permanently skipped — targets a class that does not exist — tests/test_distributed.py:14-90, 136-161, 190-211
- **Category**: correctness / testing (dead tests)
- **Problem**: Every test in `TestHeartbeatSql` (3 tests), `TestDistributedManagerInstantiation` (2 tests), and the parametrized `test_heartbeat_sql_with_different_parameters` (3 cases) does `from pyutilz.distributed import DistributedManager` inside a `try/except ImportError: pytest.skip(...)`. I verified `src/pyutilz/system/distributed.py` (the real target of the `pyutilz.distributed` alias, per `src/pyutilz/__init__.py:35`) has **no `class DistributedManager`** at all — only module-level functions `register_scraper`, `get_heartbeat_sql`, `heartbeat_scraper` and a `Container` class. `from X import DistributedManager` therefore raises `ImportError: cannot import name 'DistributedManager'` every single run, so all 8 of these tests hit `pytest.skip` unconditionally, forever.
- **Failure scenario**: The file's own docstring (lines 1-7) states its purpose is verifying "SQL injection protection (parameterized queries in heartbeat SQL)". That protection is never actually exercised: a regression that reintroduced f-string SQL interpolation into `get_heartbeat_sql()` (the real, correctly-named module-level function) would not be caught by any of these 8 tests — they never call the real function, because the wrong symbol name means they never even get past the `ImportError`. (Separately, this file is also the worst offender for the source-inspection anti-pattern below — see Medium finding.)
- **Suggested fix**: Rewrite against the real API — `from pyutilz.system.distributed import get_heartbeat_sql` and call it directly (as `TestIdentityRaceCondition` at the bottom of the same file already correctly does via `import pyutilz.system.distributed as distributed_module`).

### [High] tests/benchmark_pandaslib.py is never executed by anything — dead code masquerading as part of the suite
- **Category**: testing / organization
- **Problem**: `pyproject.toml:217` sets `python_files = ["test_*.py"]`. `tests/benchmark_pandaslib.py` does not match that glob (no `test_` prefix), so `pytest` (the whole suite, `pytest tests/`, `pytest --cov=...`) never collects a single test from it, ever. Its own docstring says the only way to run it is `python -m tests.benchmark_pandaslib`. I grepped the entire repo (workflows, `*.md`, `*.toml`, `*.cfg`, Makefile-equivalents) for `benchmark_pandaslib` and found **zero** references outside the file itself — nothing in CI, TESTING.md, or any script invokes it.
- **Failure scenario**: The file imports `from pyutilz.pandaslib import (optimize_dtypes, nullify_standard_values, get_df_memory_consumption, ensure_dataframe_float32_convertability, remove_constant_columns)` (I verified all five still exist in `src/pyutilz/data/pandaslib/{dtypes,frames}.py`, so it's not broken *today*) — but because nothing ever imports or runs this file, a future rename/removal of any of those five functions would leave this file silently broken indefinitely; nobody would notice until a contributor tries to run it by hand and gets an `ImportError`.
- **Suggested fix**: Either delete it (its content overlaps with the real, CI-collected `tests/test_benchmark.py` / `tests/test_benchmarking.py` in spirit), or move it to `_benchmarks/bench_pandaslib.py` alongside the other two standalone, deliberately-non-pytest benchmark scripts so its "not part of the automated suite" status is unambiguous instead of looking like an orphaned test file.

### [Medium] Two tautological / vacuous assertions in the same class — tests/test_web.py:22-52
- **Category**: correctness / assertion depth
- **Problem**: (a) `test_set_proxy_last_use_time` (line 42-52) ends with `assert len(last_used) > 0 or last_used == {}`. For a Python `dict`, `len(d) == 0` and `d == {}` are equivalent, so this is a logical tautology — `True` for literally any dict the function could produce, including one the function left completely untouched. (b) `test_is_rotating_proxy_true` (line 22-30) and `test_is_rotating_proxy_false` (line 32-40) both assert `result is None or isinstance(result, bool)`. I read `is_rotating_proxy()` in `src/pyutilz/web/web.py:637-644`: it only ever returns `True` or `None` (never `False`, contrary to what `test_is_rotating_proxy_false`'s name implies), and it keys off `proxy_server["PROXY_HOST"]`/`PROXY_MIN_PORT`/`PROXY_MAX_PORT` — but both tests pass in `{"http": "http://...-proxy.com:8080"}`, a dict shape the function never inspects. Both calls always fall through to `return None`, and `None` already satisfies the assertion's first clause, so the "_true" test can never actually observe the `True` branch it's named for.
- **Failure scenario**: If `is_rotating_proxy` were changed to always `raise` or always return the wrong boolean for a genuine Smartproxy-DC-gateway config, none of these three tests would notice — (a) accepts any output shape, (b)/(c) never even reach the code path they're purportedly testing.
- **Suggested fix**: For (a), assert the actual recorded value: `assert last_used[joblib_hash(proxies)]` is a recent `datetime`. For (b)/(c), pass a dict shaped like the real input (`{"PROXY_HOST": "gate.dc.smartproxy.com", "PROXY_MIN_PORT": 20000, "PROXY_MAX_PORT": 20000}`) and assert the exact expected value (`True` / `None`).

### [Medium] `except Exception: pass` around the call under test, with no assertion on the success path — tests/test_web.py:69-78
- **Category**: correctness / assertion depth
- **Problem**: `test_report_params` wraps its only call to `report_params(...)` in `try: ... except Exception: pass`, with the comment "Expected if logger not configured". There is no assertion anywhere in the test — it passes if the call succeeds silently, and it passes just as well if the call raises any exception at all.
- **Failure scenario**: `report_params` could be changed to raise `TypeError` on every call (e.g. a signature/format-string mismatch introduced by a refactor) and this test would keep passing indefinitely.
- **Suggested fix**: Drop the try/except; assert on `caplog` that the expected INFO-level record with the url/params was actually logged (the production code, `src/pyutilz/web/web.py:382-384`, is a one-line `logger.info(...)` call, trivially assertable via `caplog.at_level("INFO")`).

### [Medium] `inspect.getsource()` string-matching used as the primary regression guard in multiple files
- **Category**: testing / architecture (anti-pattern, contradicts project convention)
- **Problem**: Several tests assert on the *text* of a function/module's source rather than its *behaviour*: `tests/test_distributed.py:95-133` (`test_no_self_at_module_level`, `test_version_not_using_mtime`) and `:167-188` (`test_no_string_formatting_in_sql`, itself a weak SQL-injection guard — it only flags an f-string SQL line if the literal substring `"WHERE"` also appears on that exact line); `tests/test_system.py:17-44` (`test_no_shell_true_in_mac_uuid_extraction`), `:116-131` (`test_wintypes_import_guarded`), `:153-165` (`test_psutil_cpu_percent_called_correctly`); `tests/test_monitoring.py:16-25` (`test_executor_is_module_level`); `tests/test_parallel.py:44-53` (checks for the literal string `"@atexit.register"`) and `:150-165`.
- **Failure scenario**: A harmless, behavior-preserving refactor (e.g. renaming the module-level executor from `_TIMEOUT_EXECUTOR` to `_EXECUTOR_POOL`, or moving the `atexit.register(...)` call into a helper function so the literal substring `"atexit.register"` no longer appears verbatim in the module source) fails these tests even though nothing is actually broken. Conversely, a genuine SQL-injection regression that happens not to contain the literal substring `"WHERE"` on the same source line as the f-string (e.g. a multi-line f-string, or a `WHERE` clause built via string concatenation on a separate line) sails through `test_no_string_formatting_in_sql` undetected.
- **Suggested fix**: Replace with behavioural tests: call the function/decorator and assert on its actual runtime effect (e.g. `test_executor_is_module_level` should assert the *same* executor object is used across two calls, not grep for a variable name).

### [Medium] "Set seed" functions are tested for "doesn't crash," never for the one property a seed function exists to provide — tests/test_numbalib.py:21-73, 197-208
- **Category**: correctness / assertion depth
- **Problem**: `TestSetNumbaRandomSeed.test_accepts_integer_seed` (24-34) and `test_sets_random_state_without_error` (36-48), `TestSetRandomSeed.test_handles_none_default` (54-63) and `test_accepts_integer_seed` (65-73), plus the parametrized `test_random_seed_parametrized` (197-208, 5 seed values) all check only that `set_numba_random_seed`/`set_random_seed` don't raise, and at most that subsequently-drawn numbers have the right length/range. Two of them (lines 32, 61, 71) literally end with a bare `assert True` after a call that would already have failed the test via `pytest.fail(...)` in the `except` clause if it had raised — the `assert True` verifies nothing beyond what the `try/except` already checked.
- **Failure scenario**: If `set_numba_random_seed`/`set_random_seed` were refactored to silently ignore their `seed` argument (a real, plausible regression class — e.g. a decorator swallowing `**kwargs`), calling `set_numba_random_seed(42)` twice would produce two *different* random sequences instead of the same one, breaking reproducibility for every downstream `mlframe` caller relying on this for deterministic experiments — and none of these 5 test functions would detect it, since none of them ever call the seed function twice and compare the resulting draws.
- **Suggested fix**: Add one determinism test: `set_numba_random_seed(42); a = np.random.rand(5); set_numba_random_seed(42); b = np.random.rand(5); assert np.array_equal(a, b)`.

### [Medium] Inconsistent exception-swallowing between near-duplicate tests in sibling files — tests/test_llm_providers.py:66-73 vs tests/test_llm_factory.py:60-66
- **Category**: correctness / assertion depth
- **Problem**: Both files have a "does the `claude`/`CLAUDE`/`cc` alias resolve without an Unknown-provider error" test. `tests/test_llm_factory.py:60-66` (`test_claude_code_variants_accepted`) only catches `ValueError` and fails the test (via `pytest.fail`) if it contains "Unknown provider" — any other exception propagates and fails the test, correctly. `tests/test_llm_providers.py:66-73` (`test_case_insensitive`) does the same `ValueError`/"Unknown provider" check but then adds `except Exception: pass`, silently swallowing every other exception.
- **Failure scenario**: In an environment where `ANTHROPIC_API_KEY` is unset (the default for CI, since live tests are opt-in — see `tests/conftest.py:113-119`), `get_llm_provider("CLAUDE")` resolves the alias fine but then raises `ValueError("API key not provided...")` when constructing `AnthropicProvider` (see `AnthropicProvider.__init__`, exercised directly by `tests/test_llm_providers.py:308-315`). That's a *different* `ValueError` message, so it skips the "Unknown provider" check, falls into `except Exception: pass`, and the test passes without ever having verified alias resolution actually worked — on precisely the CI configuration this repo runs by default. The sibling test in `test_llm_factory.py` doesn't have this hole.
- **Suggested fix**: Drop the blanket `except Exception: pass` in `test_llm_providers.py:72-73` to match the (correct) pattern in `test_llm_factory.py`.

### [Medium] `gpu`, `slow`, and `integration` pytest markers are declared with specific semantics but never enforced
- **Category**: testing / marker consistency
- **Problem**: `pyproject.toml:221-226` declares 4 markers, each with a stated meaning (`gpu: marks tests requiring GPU`, `slow: ... (deselect with '-m "not slow"')`, `integration: marks tests as integration tests`, `live: ...`). Only `live` actually has enforcement logic (`tests/conftest.py:113-119`, `pytest_collection_modifyitems` auto-skips unless `--run-live`). `.github/workflows/ci.yml:70` runs plain `pytest --cov=src/pyutilz --cov-report=xml --cov-report=term-missing --cov-fail-under=60` — no `-m` filter anywhere, so `gpu`- and `slow`-marked tests run on every CI invocation regardless of the marker. `integration` has **zero** usages anywhere in `tests/` (confirmed via repo-wide grep) — it's pure dead configuration.
- **Failure scenario**: `tests/system/test_hardware_detection.py:85-97` (`test_get_nvidia_smi_info`, `@pytest.mark.gpu`) is written defensively (`if gpu_info: assert ...; else: print("[WARN] nvidia-smi not available")`), so on the GPU-less `ubuntu-latest` CI runner it always takes the `else` branch and passes trivially, checking nothing. A contributor who adds a *new* `@pytest.mark.gpu` test that asserts a real value (reasonably assuming, per the marker's documented meaning, that CI either has a GPU or skips it) would break CI the first time it runs on a GPU-less runner, since nothing filters `gpu`-marked tests out.
- **Suggested fix**: Either wire `-m "not gpu and not slow"` into the default CI run (with a separate opt-in job for `gpu`/`slow`, mirroring the `live` pattern), or remove the markers' misleading "requires GPU" framing and delete the unused `integration` marker.

### [Medium] No documented rule for tests/ subfolder placement — tests/performance/, tests/stats/, tests/system/ vs. the flat root
- **Category**: testing / organization
- **Problem**: I read `TESTING.md` in full — it documents `tests/test_meta/` in detail but never explains why `tests/performance/kernel_tuning/`, `tests/stats/`, and `tests/system/` exist as subfolders while everything else (70 files) sits flat at `tests/` root. Concretely, `src/pyutilz/system/monitoring.py`, `src/pyutilz/system/parallel.py`, and `src/pyutilz/system/distributed.py` are tested at the flat root (`tests/test_monitoring.py`, `tests/test_parallel(_extra).py`, `tests/test_distributed.py`) while `src/pyutilz/system/gpu_dispatch.py` and `src/pyutilz/system/system/*` (hardware detection) are tested inside `tests/system/` — same source package, split test placement, no stated criterion distinguishing them.
- **Failure scenario**: A contributor adding a test for a new function in `src/pyutilz/system/hardware_monitor.py` has no documented signal for whether it belongs in `tests/system/test_hardware_detection.py` (where the *existing* `UtilizationMonitor` tests for that exact module already live, lines 280-311) or a new flat-root `tests/test_hardware_monitor.py` — genuine ambiguity that produces exactly the kind of drift already visible in the `system` domain.
- **Suggested fix**: Add a short "test layout" section to TESTING.md stating the actual rule (e.g. "subfolders are for domains added after date X" or "subfolders group multi-file source subpackages"), and either move the 3 flat-root `system/` test files into `tests/system/` or state explicitly why they stay out.

### [Medium] "_extra" / "_extra2" / "_extra3" test-file fragmentation is coverage-chasing, not topical organization
- **Category**: testing / organization / maintainability
- **Problem**: 5 source modules each have 2-4 same-module test files whose own docstrings confirm they were added purely to close coverage gaps, not to group related behavior: `tests/test_system_extra3.py:1` ("Tests targeting uncovered lines in system.py (extra3)"), `tests/test_pandaslib_extra2.py:1` ("Additional coverage tests ... targeting uncovered lines"), `tests/test_polarslib_extra2.py:1-4` (docstring is literally a list of uncovered line-number ranges: "98-154, 231, 287, 289, 293, 326-328..."), `tests/test_pythonlib_extra2.py:1` ("Extra tests ... covers specific uncovered lines"), `tests/test_strings_extra2.py:1` ("Extra tests ... covers uncovered lines"). None of the "_extra"/"_extra2"/"_extra3" files carry any docstring cross-reference back to the base file.
- **Failure scenario**: A contributor wanting to add a test for, say, `pyutilz.data.pandaslib.dtypes.optimize_dtypes` must grep across `test_pandaslib.py`, `test_pandaslib_extra.py`, and `test_pandaslib_extra2.py` (I checked — no name-level duplication today, but nothing prevents it going forward) before knowing whether coverage already exists or where to add it; the 4-way split for `system.py` (`test_system.py`/`_extra`/`_extra2`/`_extra3`) is the worst case.
- **Suggested fix**: Either merge each family into one file per source module, or rename by *topic* (e.g. `test_system_gpu.py`, `test_system_wmi.py`) instead of by *when the coverage gap was noticed*.

### [Medium] `sqlite.py` helpers have zero behavioural test coverage — only a facade re-export identity check — tests/test_db_extra.py:272-311
- **Category**: testing / coverage gap
- **Problem**: `src/pyutilz/database/db/sqlite.py` (78 lines) defines `ensure_db_tables_created` and `insert_sqllite_data`. I grepped the entire `tests/` tree for both names: the only hits are `tests/test_db_extra.py:284-285` (membership check that the facade `hasattr` exposes the name) and `:311` (`assert db.insert_sqllite_data is sqlite.insert_sqllite_data`, an identity check against object aliasing from the `db.py` → subpackage split). Neither function is ever *called* with a real or fake `sqlite3.Connection`/`Cursor` anywhere in the suite.
- **Failure scenario**: `insert_sqllite_data` could build a malformed `INSERT` statement (wrong column count, wrong parameter binding) or `ensure_db_tables_created` could silently no-op instead of creating the schema, and no test in the repo would notice — both checks above only confirm the *name* is reachable, never that the *function* behaves correctly.
- **Suggested fix**: Add a real test using `sqlite3.connect(":memory:")`: create a table via `ensure_db_tables_created`, insert rows via `insert_sqllite_data`, and assert the row count/values via a follow-up `SELECT`.

### [Low] Test freezes a source-code typo as the "correct" contract — tests/system/test_hardware_detection.py:288-311
- **Category**: testing / maintainability
- **Problem**: `test_utilization_monitor_basic_usage` asserts `"cpu_utilizaton_percent" in avg_util` (line 310) and indexes `avg_util['cpu_utilizaton_percent']` (line 311) — "utilizaton" is missing the second "i" (should be "utilization"). I confirmed this exact misspelling is the real, shipped public dict key in `src/pyutilz/system/hardware_monitor.py:43,186,200` (including in the module's own doctest example at line 43), so the test is faithfully mirroring production, not itself buggy.
- **Failure scenario**: If a future PR fixes the production typo (renaming the key to `cpu_utilization_percent`), this test — and the doctest at `hardware_monitor.py:43` — will fail, making the correct fix look like a regression rather than a welcome cleanup.
- **Suggested fix**: Out of this audit's scope to fix the production typo (that's a public-API-naming call for the source-code owner), but worth flagging together since the test is what will need to change in lock-step.

### [Low] Parametrization opportunities — near-duplicate test bodies differing only by a literal
- **Category**: testing / maintainability
- **Problem**: `tests/test_llm_providers.py` `TestAnthropicProvider.test_pricing_sonnet` / `test_pricing_opus` / `test_pricing_fallback` (lines 284-306) are three near-identical bodies (`p.model = "..."; inp, out = p._get_pricing(); assert inp == X; assert out == Y`) differing only in the model string and expected tuple. Same pattern in `TestGeminiProvider.test_pricing_flash_lite` / `test_pricing_fallback` (346-360).
- **Failure scenario**: Not a bug, just maintenance overhead — adding a 4th pricing tier requires copy-pasting a whole method instead of adding one row to a parametrize table.

### [Low] `_provider_cache` singleton not cleared by every test that populates it
- **Category**: testing / flaky-risk (shared global state)
- **Problem**: `tests/test_llm_factory.py`'s `TestGetLlmProvider` correctly clears the module-level `_provider_cache` singleton in `setup_method`/`teardown_method` (lines 42-46). `tests/test_llm_providers.py:TestFactory.test_case_insensitive` (line 66-73) also calls `get_llm_provider("CLAUDE")`, which — when an API key happens to be present in the environment (e.g. a contributor's local `.env` used for live-test setup per TESTING.md) — populates the same shared `_provider_cache` dict, but this file never clears it before or after.
- **Failure scenario**: Currently harmless only because `test_llm_factory.py` defensively clears the cache on entry to each of its own tests regardless of prior state. It's fragile: any future test added to either file that asserts on `_provider_cache` contents without first clearing it (the way `test_unhashable_kwarg_bypasses_cache_but_is_tracked_for_cleanup`, lines 81-100, inspects `_provider_cache.values()`) would become order-dependent under `pytest-randomly` (registered in `conftest.py`, confirmed via the thinc-seed-shim comment), since it may or may not see an entry left behind by `test_llm_providers.py`.
- **Suggested fix**: Add the same `setup_method`/`teardown_method` `_provider_cache.clear()` pattern to `tests/test_llm_providers.py:TestFactory`.

### [Low] Four differently-purposed, undocumented "benchmark" surfaces with confusable names
- **Category**: testing / organization
- **Problem**: I read all four in full. `_benchmarks/bench_classify_column_types.py` and `_benchmarks/bench_get_columns_of_type.py` (repo root, outside `tests/`) are standalone `__main__`-guarded scripts with hard-coded identity assertions comparing an old vs. new implementation, run manually (`python _benchmarks/bench_*.py`) — genuinely distinct from pytest, and clearly labelled as such. `tests/benchmark_pandaslib.py` is *also* a standalone, non-pytest, `__main__`-guarded script (see the High finding above: it is never collected or run by anything) but its location *inside* `tests/` gives the opposite impression. `tests/test_benchmark.py` (74 lines) is real, CI-collected pytest coverage for `pyutilz.performance.kernel_tuning.benchmark` (`time_backend`/`benchmark_backends`). `tests/test_benchmarking.py` (346 lines) is real, CI-collected pytest coverage for the entirely different `pyutilz.dev.benchmarking` module (`benchmark_algos_by_runtime`/`sweep_backend_grid`). None of this is explained anywhere in TESTING.md.
- **Failure scenario**: `test_benchmark.py` and `test_benchmarking.py` differ by one character in the filename and test two unrelated modules — a contributor grepping for "the benchmark test" or fixing a failure reported by CI under one of these names has a real chance of editing the wrong file.
- **Suggested fix**: Rename for clarity (e.g. `test_kernel_tuning_benchmark.py` and `test_dev_benchmarking.py`), and add one sentence to TESTING.md distinguishing all four surfaces.

## Things done well

- **`tests/performance/kernel_tuning/test_cache.py` and `test_cache_concurrency.py`** are the strongest files in the suite: deliberate singleton/global-state hygiene (fixtures clear `_TUNED_THIS_PROCESS`, `_DEFAULT_INSTANCE`, `hw_fingerprint.cache_clear()` on **both** entry and exit — e.g. `TestLoadOrCreate.setup_method`/`teardown_method`, lines 433-438), real subprocess-isolated multi-process tests for the immutable-file cache redesign (D1 lost-update, stale-marker steal via dead-PID vs. mtime-expiry vs. fresh-empty-marker, single-winner claim races), and comments that document *why* a timeout value was chosen (`test_cache_concurrency.py:57-61`, the 300s subprocess timeout is justified against real "postgres-saturated disk" cold-import cost, not a magic number).
- **`tests/stats/test_normality.py`** is statistically rigorous rather than brittle: fixed RNG seeds, false-positive-rate tolerance bands calibrated against the known alpha (not exact-value pinning), and scipy-parity assertions to float tolerance.
- **`tests/system/test_gpu_dispatch.py`** mocks only the true CUDA-availability boundary (`is_cuda_available`) and lets all real arithmetic run, with exact pinned per-compute-capability shared-memory budgets and threads-per-block heuristics (e.g. `test_opt_in_specific_values_per_cc`) — real value assertions, not existence checks.
