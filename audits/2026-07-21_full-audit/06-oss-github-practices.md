# OSS / GitHub Project-Health Audit — pyutilz

## Summary

Covered repo-root plumbing and all of `.github/` for pyutilz: read every workflow file (`ci.yml`, `black-filtered.yml`, `docs.yml`, `mypy-full.yml`, `numba-coverage.yml`, `publish.yml`) and `dependabot.yml` in full, plus `README.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, `LICENSE`, `pyproject.toml` (classifiers/urls/version), `.pre-commit-config.yaml`, `.gitattributes`, `.yamllint`, and `docs/index.md` for cross-checks. The CI/CD plumbing itself is unusually mature (SHA-pinned Actions, least-privilege `permissions:` blocks, Trusted Publishing, zizmor/actionlint wired into both CI and pre-commit) — the real gaps are in the community-health layer (no SECURITY.md, CODE_OF_CONDUCT.md, issue/PR templates, CODEOWNERS, CITATION.cff — all confirmed absent by a full-repo search) and in a handful of cross-file consistency defects (branch-name mismatch, shared-workflow version drift, a semver-relevant CHANGELOG entry) that no existing gate catches because none of them are lint-shaped. Findings: 1 High, 4 Medium, 7 Low.

## Findings

### [High] Tag-triggered PyPI release has no in-repo gate against an arbitrary/unreviewed commit — .github/workflows/publish.yml:7-10,44-57
- **Category**: security / OSS-practice (release process)
- **Problem**: `publish.yml` triggers on `push: tags: ["v*"]` (any branch, any commit) with no `workflow_dispatch`-only requirement and no check that the tagged commit is an ancestor of `master`. The `build` job (lines 16-42) runs unconditionally for any such tag — no test suite is invoked, only `python -m build` + `twine check` (metadata-only). The `publish` job (lines 44-57) uses `environment: pypi` (line 47) with Trusted Publishing (OIDC, `id-token: write`), which is the *correct* mechanism per se (no long-lived token to leak), but whether it actually stops an unreviewed tag from reaching PyPI depends entirely on whether the `pypi` GitHub Environment has "required reviewers" protection configured — a setting that lives in repo configuration, not in this file, and nothing in the tree gives evidence it's set.
- **Failure scenario**: Any collaborator with ordinary "Write" repo access (not necessarily Admin/Maintain) can, from any local clone, do `git tag v99.0.0 <any-commit-they-can-push> && git push origin v99.0.0` — including a commit on a throwaway branch that never went through PR review or CI. `publish.yml` fires, builds an sdist/wheel from that exact commit, `twine check` passes (it validates metadata, not code), and `pypa/gh-action-pypi-publish` ships it to PyPI as the new `pyutilz` release, without ever running `pytest` and without a human approval step unless the `pypi` environment has reviewer protection turned on separately in GitHub's UI.
- **Suggested fix**: Configure the `pypi` GitHub Environment with required reviewers (Settings → Environments → pypi → protection rules) and/or a tag-protection rule restricting who can push `v*` tags (Settings → Tags), and/or add a workflow-level check (e.g. `git merge-base --is-ancestor <tag> origin/master`) before the `build`/`publish` jobs proceed.

### [Medium] The same shared reusable-workflow repo is pinned to three different, unexplained versions — .github/workflows/black-filtered.yml:20, ci.yml:85,91,103, mypy-full.yml:27, docs.yml:33, .pre-commit-config.yaml:263
- **Category**: OSS-practice / maintainability
- **Problem**: `fingoldo/py-ci-shared` is consumed at three distinct pinned revisions with no comment anywhere explaining the split:
  - `.pre-commit-config.yaml:263` → `rev: v1.0.0` (the `mypy-full-manual` hook)
  - `ci.yml:85,91,103` / `mypy-full.yml:27` / `docs.yml:33` → all pin `f26052fd4fee0d667aa6f4c3290d77e7ca6c1c06 # v1.2.0`
  - `black-filtered.yml:20` → pins `1fd408df105f2a7d22d36a239d94a39ef888305a # v1.2.1`
  This is exactly the class of drift the project's own `pyproject.toml` comment on `ruff==0.15.20` (line 183-185) warns about ("an open range here is exactly how mlframe's prior v0.8.6 pre-commit-vs-CI drift went unnoticed for a long time") — just one layer up, at the shared-workflow-repo level instead of the pip-package level.
- **Failure scenario**: If `v1.2.1` fixed a real bug in the shared `black_filtered_apply` logic relative to `v1.2.0`, CI's `lint-blocking`/`lint-advisory`/`mypy-full.yml`/`docs.yml` jobs are still running the pre-fix `v1.2.0` code while only the Black workflow got the fix. Locally, `pre-commit run --hook-stage manual mypy-full-manual` exercises `v1.0.0` of the shared mypy wrapper while CI's `mypy-full.yml` exercises `v1.2.0` — if the wrapper's argument handling changed across those two releases, a contributor's local "clean" manual run can diverge from what CI actually enforces, for reasons unrelated to their code.
- **Suggested fix**: Bump every reference to the same current SHA/tag of `py-ci-shared`, and if a deliberate version split is ever needed again, add the same kind of one-line rationale comment this repo already uses elsewhere for intentional version pins.

### [Medium] `pyproject.toml`'s Changelog URL uses a branch name inconsistent with every other hardcoded branch reference in the repo — pyproject.toml:204 vs .github/workflows/docs.yml:10-12,35 and docs/index.md:8,100
- **Category**: docs / OSS-practice
- **Problem**: `pyproject.toml:204` sets `Changelog = "https://github.com/fingoldo/pyutilz/blob/main/CHANGELOG.md"` — branch `main`. Every other hardcoded branch reference in the repo instead uses `master`: `docs.yml` only triggers the GitHub Pages deploy on `push: branches: [master]` (line 10) and passes `default-branch: master` as an explicit input to the reusable workflow (line 35, with its own comment about `Enable Pages ... to publish`); `docs/index.md` hardcodes `blob/master/LICENSE` twice (lines 8 and 100). If the repo's actual default/only branch were `main`, GitHub Pages would never redeploy on ordinary pushes — `docs.yml`'s single-branch specificity is the strongest signal of which branch is real, and it says `master`.
- **Failure scenario**: `pip install pyutilz` publishes this exact URL on the package's PyPI project page under "Changelog" (`project.urls` values are rendered verbatim by PyPI's sidebar). If no `main` branch exists in the repository (as every other reference implies), every visitor who clicks "Changelog" from PyPI gets a 404.
- **Suggested fix**: Change `pyproject.toml:204` to `blob/master/CHANGELOG.md` (or, if `main` is in fact correct, fix `docs.yml` and `docs/index.md` instead) — pick one branch name and make every hardcoded reference agree.

### [Medium] No SECURITY.md and no reachable vulnerability-reporting contact anywhere in the repo — repo root (absent), CONTRIBUTING.md:214-220
- **Category**: OSS-practice / security-process
- **Problem**: A full-repo search (`find . -iname "SECURITY*"`) confirms no `SECURITY.md` exists at root or under `.github/`. CONTRIBUTING.md's "Reporting Vulnerabilities" section (lines 216-220) says: *"email security concerns to the maintainers or use GitHub's private vulnerability reporting"* — but no email address appears anywhere in the tracked repo (checked README.md, CONTRIBUTING.md, `pyproject.toml`'s `authors` field, LICENSE); `pyproject.toml:9` declares `authors = [{name = "Anatoly Alexeev"}]` with no `email` key. GitHub also specifically uses a `SECURITY.md` file to populate a repo's "Security policy" tab; without one, that tab shows no custom guidance regardless of whether Private Vulnerability Reporting is separately toggled on in repo settings (not visible from the working tree either way).
- **Failure scenario**: A researcher who finds, say, a bypass in `validate_sql_identifier()` (the function README.md itself highlights under its own "Security" section, line 253) follows CONTRIBUTING.md's instruction to "email the maintainers" and has no address in the repo to send it to; they fall back to "GitHub's private vulnerability reporting," which only works if that repo setting has separately been enabled — something nothing in this codebase evidences.
- **Suggested fix**: Add a `SECURITY.md` (root or `.github/`) with a real reporting channel (an email address, or an explicit statement that GitHub private reporting is enabled) and a supported-versions policy.

### [Medium] CHANGELOG's `[Unreleased]` section already contains a public-API removal filed as a routine "Fixed," not a breaking change — CHANGELOG.md:8,24-26 vs header line 6
- **Category**: OSS-practice / semver
- **Problem**: CHANGELOG.md states "this project adheres to Semantic Versioning" (line 6), and the package is already released at `1.0.0` (line 32, `pyproject.toml:7`). The `[Unreleased] → Fixed` section (line 25) lists: *"14 accidentally-public leaked-import symbols removed from `pyutilz.image` / `logginglib` / `numpylib` / `parallel` (never intended API; real homes documented in each symbol's replacement)."* SemVer is defined over the actually-importable public surface, not author intent — any consumer who did `from pyutilz.image import <leaked_symbol>` post-1.0.0 has working code today that the next release breaks with `ImportError`. That's a breaking change by SemVer §8 ("any backwards incompatible changes ... MUST be released as a major version"), yet it's filed under `Fixed` in the same `[Unreleased]` bucket as several genuinely additive `Added` features (new `code_audit` module, OpenRouter provider methods, `thinking=` effort strings, etc.) — meaning whoever cuts the next release naturally reads "mostly Added" and tags it `1.1.0`, silently mislabeling a breaking release as minor.
- **Failure scenario**: Maintainer runs `git tag v1.1.0` per the section's "Added"-dominated content. Downstream projects that pinned `pyutilz>=1.0,<2.0` (a standard, SemVer-trusting pin) auto-upgrade via their own dependency bot; any of them importing one of the 14 removed leaked symbols breaks in production, with no major-version forcing function they could have used to defer the upgrade.
- **Suggested fix**: Either restore the 14 symbols as deprecated re-exports (`DeprecationWarning`) for one more major version before removing them, or tag the next release `2.0.0` and record the removal explicitly as `### Removed` / `BREAKING CHANGE` rather than folding it into `### Fixed`.

### [Low] No CODE_OF_CONDUCT.md — only a 3-sentence inline paragraph in CONTRIBUTING.md — CONTRIBUTING.md:5-7
- **Category**: OSS-practice
- **Problem**: Confirmed no standalone `CODE_OF_CONDUCT.md` exists anywhere in the tree. CONTRIBUTING.md's "Code of Conduct" section is three sentences with no enforcement contact, no scope, and no consequences — most CoC frameworks (e.g. Contributor Covenant, which GitHub's own community-profile checklist expects) call for a dedicated file so it's independently linked from the repo's "Community" tab rather than buried inside CONTRIBUTING.md.
- **Failure scenario**: A contributor experiencing harassment in a PR thread has no documented "who do I report this to" — the current text names no contact at all, only "be respectful."
- **Suggested fix**: Adopt a standard template (e.g. Contributor Covenant) as `CODE_OF_CONDUCT.md`, with a real enforcement contact, linked from both CONTRIBUTING.md and README.md.

### [Low] No `.github/ISSUE_TEMPLATE/` or `PULL_REQUEST_TEMPLATE.md` despite detailed inline guidance already written — repo root/.github (absent), CONTRIBUTING.md:11-29,282-291
- **Category**: OSS-practice
- **Problem**: CONTRIBUTING.md spells out exactly what a bug report (lines 13-20) and enhancement suggestion (lines 24-29) should contain, and separately gives an 8-item PR checklist (lines 282-291) — but none of it is wired into an actual `.github/ISSUE_TEMPLATE/*.md` (or issue-forms `.yml`) or `.github/PULL_REQUEST_TEMPLATE.md`. GitHub's "New issue" / "New pull request" UI never surfaces this structure; it's only discoverable if the contributor separately reads CONTRIBUTING.md first.
- **Failure scenario**: A first-time contributor clicks "New issue," gets a blank text box, and files a report missing the environment/traceback details CONTRIBUTING.md asks for — because the guidance that exists was never wired into the form that actually collects the report.
- **Suggested fix**: Add `.github/ISSUE_TEMPLATE/bug_report.md` and `feature_request.md` (or an issue-forms `config.yml`), plus `.github/PULL_REQUEST_TEMPLATE.md`, mirroring the content already written in CONTRIBUTING.md.

### [Low] No `CITATION.cff` despite the package being classified for a research audience — pyproject.toml:22,37
- **Category**: OSS-practice
- **Problem**: `pyproject.toml` classifies the package under `Intended Audience :: Science/Research` (line 22) and `Topic :: Scientific/Engineering` (line 37), but no `CITATION.cff` exists anywhere in the repo (confirmed via full-repo search). GitHub natively renders a "Cite this repository" button from `CITATION.cff`; without it there's no canonical, machine-readable citation for academic users building on e.g. the `KernelTuningCache` methodology or the numba-accelerated similarity module referenced in the README's own examples.

### [Low] CHANGELOG duplicates a historical (already-released) fix inside the `[Unreleased]` section — CHANGELOG.md:26 vs 51-52
- **Category**: docs / OSS-practice
- **Problem**: `[Unreleased] → Fixed` (line 26) lists *"SQL-injection and command-injection risks in `db.py` / `system.py` (historical, 0.90)."* The identical fix is already independently recorded five sections down under `[0.90]` (lines 51-52: *"SQL-injection vulnerabilities in `db.py` (6 locations) and command-injection risk in `system.py`."*). ## Things done well

- Every third-party GitHub Action across all 6 workflows is pinned to a full commit SHA with a version-number comment (`actions/checkout@34e114...  # v4`, `codecov/codecov-action@0fb717...  # v5`, etc.) — no floating `@v4`-style tags anywhere, closing off the most common Actions supply-chain risk.
- Explicit least-privilege `permissions:` blocks at both workflow and job level in every file, including a genuinely useful documented gotcha in `docs.yml` (lines 19-25) about reusable-workflow permission ceilings that would otherwise be a confusing failure to debug.
- `publish.yml` uses PyPI Trusted Publishing (OIDC via `id-token: write`), with zero long-lived API token stored as a secret, and correctly isolates `id-token: write` to only the `publish` job (the `build` job stays `contents: read`-only).
- zizmor (GitHub Actions security linter) and actionlint are wired into both CI (`lint-blocking` job, `run-zizmor: true`) and pre-commit (`zizmor-blocking`, `actionlint` hooks), catching the floating-tag/permission-escalation/template-injection class of bug proactively rather than relying on manual review.
- `dependabot.yml` has clear, dated inline comments explaining a non-obvious config choice (pip ecosystem throttled to `open-pull-requests-limit: 0` for Actions-minutes budget reasons) — good "explain the surprising config" discipline that most repos skip.
- CHANGELOG.md faithfully follows Keep a Changelog format with real per-release substance (specific function names, measured perf numbers, migration notes), not just "misc fixes."

## Investigated, not an issue

- Checked whether any workflow's `permissions:` block might be missing or too broad, and whether calling a reusable workflow could silently escalate privileges beyond what the caller grants: confirmed every workflow declares `contents: read` at minimum and only escalates (`pages: write`/`id-token: write`) where functionally required; `docs.yml` even documents the reusable-workflow permission-ceiling behavior inline. Not a gap.
- Checked whether README's CI/MyPy/Black/numba-coverage badge URLs reference a workflow file or job name that doesn't exist: all four resolve correctly against the actual `name:` fields (`CI`, `Black`) and filenames (`mypy-full.yml`, `numba-coverage.yml`) in `.github/workflows/`.
- Checked whether the `[main, master]` dual-branch triggers in `ci.yml`/`mypy-full.yml`/`black-filtered.yml` were themselves a bug: they're not — listing both branches in a trigger is a defensively-safe no-op if only one exists (the extra name just never fires), which is a materially different situation from the `docs.yml`/`pyproject.toml` mismatch (a single, specific, wrong branch reference), so only the latter is reported as a finding.
- Considered whether dependabot's pip ecosystem being fully throttled to `open-pull-requests-limit: 0` (dependabot.yml:23) was neglect: it's explicitly dated (2026-07-10) and rationalized in an inline comment (Actions-minutes budget across the account), not an oversight.
- Checked `.pre-commit-config.yaml`'s mixed pinning styles (SHA for Actions-adjacent tooling vs. tag `rev:` for pre-commit hooks like `detect-secrets`, `actionlint-py`, `codespell`): tag-based `rev:` is the standard, expected convention for pre-commit's own hook-repo model (distinct from GitHub Actions `uses:` pinning, which is what zizmor and this project's SHA-pinning convention specifically target) — not a gap of the same class as the GH Actions SHA-pinning practice.
