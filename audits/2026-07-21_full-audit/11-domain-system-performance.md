# pyutilz audit — Domain: system (parallel/distributed/GPU dispatch/monitoring/probing) + performance (kernel tuning cache)

## Summary

Covered every file in the assignment in full: `system/__init__.py`, `config.py`, `distributed.py`,
`gpu_dispatch.py`, `hardware_monitor.py`, `monitoring.py`, `parallel.py`, `scheduling/__init__.py`,
`scheduling/prefect.py`, `system/system/*` (`__init__`, `_common`, `fsutils`, `memory`, `misc`,
`probing`, `sysinfo`), and all of `performance/` (`__init__`, `kernel_tuning/__init__`,
`benchmark.py`, `cache/{__init__,cache_base,cache_class,cache_hooks,region_matching}.py`,
`code_versioning.py`, `registry.py`, `remote.py`). One out-of-scope file
(`text/strings/configfiles.py`) was opened to verify a `distributed`/`prefect` dependency.

The CHANGELOG's core claim about the v3 kernel-tuning cache — immutable per-file writes, atomic
tempfile+`os.replace`, no filelock, no lost updates from concurrent local writers — **holds up**
under direct code reading; that part of the rewrite is genuinely solid engineering (see "Things
done well"). Two real gaps were found adjacent to that claim: the legacy-monolith migration lock
has no crash-recovery (an orphaned marker after a killed process permanently disables migration),
and the *remote* (S3) sharing layer can silently lose other hosts' tunings because its
read-through gate is all-or-nothing, not per-kernel. Separately, `benchmark.py`'s per-backend
timing has zero exception isolation, so one broken/incompatible variant poisons an entire tuning
sweep rather than being skipped — the opposite of "graceful degradation." Outside the cache, the
`system/system/*` probing code has a real dead-branch bug (`"Mac"` vs the real `platform.system()`
value `"Darwin"`) that silently drops a field `distributed.py` depends on, plus a recurring
unguarded-`None` bug pattern around `psutil.cpu_count(logical=False)` that the codebase's own
sibling helper already documents but doesn't reuse, and a genuine concurrency-contract violation in
`distributed.py` where the identity lock's own docstring promises protection the code doesn't
deliver.

Findings below, most severe first: 5 High, 7 Medium, 7 Low.

## Findings

### [High] `benchmark_backends`/`time_backend` have zero per-variant exception isolation — src/pyutilz/performance/kernel_tuning/benchmark.py:178-223
- **Category**: correctness / error-handling (the "handles a variant that errors out gracefully" requirement from the assignment)
- **Problem**: `time_backend()`'s warmup loop (`for _ in range(warmup): fn(*make_inputs())`, lines 178-179) and its timed loop (`fn(*args)`, line 190) call the backend callable with no `try/except` at all. `benchmark_backends()` (lines 212-223) is a bare dict comprehension calling `time_backend` once per backend, also with no isolation. If any ONE backend in the dict raises (a very realistic occurrence for GPU kernel variants — e.g. `cupy.cuda.memory.OutOfMemoryError` on a large swept input size, a shared-memory-limit error from an under-provisioned block size, an unsupported compute capability), the exception propagates out of `benchmark_backends()` entirely.
- **Failure scenario**: A project registers `kernel_tuner(kernel_name="k", tuner=my_sweep, ...)` where `my_sweep()` calls `benchmark_backends({"cpu": cpu_fn, "gpu_shared": gpu_shared_fn, "gpu_atomic": gpu_atomic_fn}, make_inputs)`. `cpu_fn` and `gpu_shared_fn` both run fine and would produce good timings; `gpu_atomic_fn` raises for one of the swept input sizes (e.g. OOM). The exception unwinds out of `my_sweep()` (the tuner). `cache_class.py`'s `_run_tuner` (lines 734-742) catches it at `logger.debug` (invisible at default INFO/WARNING log levels) and returns `None`. Back in `get_or_tune` (lines 723-726) and in `_spawn_async_sweep` (lines 779-782), `if regions:` is False, so `cache.update(...)` is never called — **none** of the measurements are persisted, including the two backends that worked perfectly. The kernel is now permanently stuck on `fallback` (often the slow default) for the lifetime of the cache, until a human notices via DEBUG logs and fixes the one broken variant. This directly defeats the stated purpose of the cache ("picks the fastest implementation... per input-size region").
- **Suggested fix**: Wrap each `fn(*args)` call (warmup and timed) in `try/except`, treat a raising variant as `float("inf")` (or drop it) for that region, and log at WARNING (not DEBUG) so the tuner can still record valid timings for the variants that worked instead of discarding the whole sweep.

### [High] S3 remote backend: partial local view causes cross-host lost updates — src/pyutilz/performance/kernel_tuning/cache/cache_class.py:271-316, 356-401
- **Category**: concurrency correctness / distributed correctness
- **Problem**: `_load()`'s read-through-from-remote path is gated by `if not kernels and self._remote is not None:` (line 294) — a boolean on whether the **local** directory scan found *any* kernel at all, not a per-kernel check. Once a host has even one locally-tuned kernel from a prior session, a fresh process's `_load()` never consults the remote store for kernels that exist *only* remotely (tuned by a different host sharing the same `hw_fingerprint`). `_persist_kernel`'s remote write-through (lines 379-388) then calls `self._remote.write(hw_fingerprint(), self._remote_payload())`, and `_remote_payload()` (lines 390-401) serializes `self._loaded["kernels"]` — i.e. exactly and only what this process's local view contains.
- **Failure scenario**: Host A (fresh) tunes kernel `foo`; local + remote both get `{"foo": ...}`. Host B (same `hw_fingerprint`, e.g. identical CPU+GPU model, per the module's own stated multi-host sharing use case) already has a *different* kernel `bar` tuned locally from an earlier session, so on process start its `_load()` finds `kernels = {"bar": ...}` locally (non-empty) and **skips the remote read-through entirely** — it never learns about `foo`. When Host B's process later tunes/updates `bar` again (or any other kernel), `_persist_kernel` pushes `_remote_payload()` = `{"bar": ...}` to the shared S3 object, **overwriting and permanently deleting `foo`** from the object every other host (including Host A, on its next fresh process) would read-through from. This is the primary scenario the `remote.py` module exists for ("A team can share HW-calibrated tunings... one object per hw_fingerprint"), and it silently regresses.
- **Suggested fix**: Merge with the remote payload (union of remote + local kernel sets, remote entries not shadowed locally passed straight through) every time a write-through happens, not only when the local store is completely empty; or gate the read-through per-kernel-name rather than on the whole local set being empty.

### [High] Legacy-cache migration claim file has no crash-recovery — permanently disables future migration — src/pyutilz/performance/kernel_tuning/cache/cache_class.py:181-227
- **Category**: concurrency correctness / robustness
- **Problem**: `_migrate_legacy()` claims `.migrate.INPROGRESS` via `os.open(claim, os.O_CREAT|os.O_EXCL|os.O_WRONLY); os.close(fd)` (lines 196-198) — an **empty** file, no payload. The claim is only removed in the `finally` of the *second* `try` block (lines 204-227), which covers reading/parsing/persisting the legacy monolith. Unlike the sweep `INPROGRESS` marker (`_try_create_marker`/`_maybe_steal_marker`, lines 836-935), which embeds `pid` + `start_ts` and has an explicit steal-if-dead-or-stale mechanism, there is **no** staleness/steal logic anywhere for `.migrate.INPROGRESS` — confirmed by grep: `_pid_alive` and `_sweep_budget_seconds` are referenced only inside the sweep-marker code path, never near migration.
- **Failure scenario**: A process is killed (OOM-killer, `kill -9`, container eviction, power loss — all realistic during a large legacy-monolith migration, since `_persist_kernel` does synchronous file I/O with retries per kernel) after creating `.migrate.INPROGRESS` but before the `finally: os.remove(claim)` runs. The claim file is now orphaned forever. Every subsequent process on that host calls `_migrate_legacy()` exactly once (`self._migrated = True` gate, line 190) and gets `FileExistsError` at the `os.open(...)` call → immediately `return`s (line 199-200) without migrating, without retry, and without logging anything above DEBUG-adjacent noise. The legacy `<fp>.json` monolith is never renamed to `.migrated` and never merged into the v3 store — any kernels that existed only in it become permanently unreachable via `lookup()` until a human manually deletes `.migrate.INPROGRESS`.
- **Suggested fix**: Reuse the same pid+start_ts+steal pattern already implemented for sweep markers (`_try_create_marker`/`_maybe_steal_marker`) for the migration claim.

### [High] `platform.system() == "Mac"` never matches on real macOS; `os_serial` silently never set — src/pyutilz/system/system/sysinfo.py:152 (contrast: src/pyutilz/system/system/probing.py:401, 506)
- **Category**: correctness / cross-platform dead branch
- **Problem**: `get_system_info()`'s sensitive-info block checks `elif current_system == "Mac":` (line 152) to run the `ioreg | grep UUID` pipeline for a real hardware UUID. `platform.system()` returns `"Darwin"` on macOS, never `"Mac"` — confirmed by this exact package's own correct usage two files over: `probing.py:401` (`check_large_pages_support`) and `probing.py:506` (`get_power_plan`) both correctly check `current_system == "Darwin"`. The `if/elif` chain in `sysinfo.py` (Windows / Linux / Android / Mac) has no trailing `else`, so on real macOS **none** of the branches fire.
- **Failure scenario**: On an actual Mac, `get_system_info(return_sensitive_info=True, ...)` (the codepath `distributed.py`'s `register_scraper()` uses by default) never runs the ioreg UUID extraction, and — more importantly — `info["os_serial"]` is **never set at all** (it is only ever assigned inside one of the Windows/Linux/Android/Mac branches; there is no macOS-reaching branch and no fallback). `distributed.py:107` builds `fields = "host_name,os_machine_guid,os_serial"` and passes `source=info` straight into `db.db_command(...)` for node registration — a completely missing dict key there is a real risk of a `KeyError` or a silently-`NULL` field depending on how that helper resolves fields (outside this audit's file scope to confirm further, but the field is unconditionally expected). `info["os_machine_guid"]` at least still gets the generic `uuid.getnode()`-based fallback (line 118, set unconditionally before the branch), so only `os_serial` is affected.
- **Suggested fix**: `elif current_system == "Darwin":` to match the rest of the package; add a trailing `else:` that at least sets `info["os_serial"] = info.get("os_machine_guid", "")` for any unrecognized platform, matching the pattern already used inside each existing branch's own `except`.
- **Secondary, lower-confidence note**: the `elif current_system == "Android":` branch (line 144) is very likely similarly dead — `platform.system()` on Android (Termux and most CPython-on-Android builds) reports `"Linux"` (it comes from `os.uname().sysname`, and Android's kernel is Linux), not `"Android"`. Flagging as lower confidence than the Mac/Darwin case since there's no in-repo cross-reference to confirm it the same way, but it is well-documented general Python behavior.

### [High] `distributed.py`: `_container.node_id` check-then-act race contradicts the lock's own docstring — src/pyutilz/system/distributed.py:39-42, 72-73, 94-127, 141-143
- **Category**: concurrency correctness (race condition)
- **Problem**: The module declares `_identity_lock` with the comment "Guards concurrent reads/writes of the scraper-node identity globals above (`_container`, `pid`, `m_app_name`, `m_scraper_name`, `m_version`, `m_ip`)" (lines 39-42). In practice, `pid` (lines 72-73) and `m_app_name`/`m_scraper_name`/`m_version`/`m_ip` (lines 94-95) *are* written under the lock, and `get_heartbeat_sql` correctly reads everything, including `_container.node_id`, under the lock (lines 141-143). But `register_scraper()`'s own check-then-act on `_container.node_id` — `if _container.node_id is None:` (line 97) through the DB `select`/`insert` calls that eventually set it via `fetch_into=_container` (lines 108-113), all the way to the final `return _container.node_id` (lines 126-127) — runs **entirely outside `_identity_lock`**, despite `_container` being explicitly named in the lock's own docstring as protected.
- **Failure scenario**: Two threads call `register_scraper()` concurrently before the first registration completes. Both observe `_container.node_id is None` (line 97), both proceed to call the (slow, I/O-bound) `system.get_system_info(only_stats=False)` and both attempt the `db.db_command("select"/"insert", "nodes", ...)` sequence concurrently — redundant/racy registration attempts against the `nodes` table (at best wasted work, at worst a duplicate-insert conflict depending on the table's constraints, which are outside this file's scope to verify). This is exactly the class of bug the lock exists to prevent, per its own comment, but the actual mutating code path never acquires it.
- **Suggested fix**: Wrap the `if _container.node_id is None: ... db_command(...) ...` block (lines 97-127) in `with _identity_lock:` the same way the other identity globals are protected.

### [Medium] `psutil.cpu_count(logical=False)` can return `None`; three call sites crash instead of degrading — src/pyutilz/system/parallel.py:136, 190; src/pyutilz/system/system/misc.py:194 (contrast: src/pyutilz/system/parallel.py:149-158)
- **Category**: correctness / edge-case
- **Problem**: `parallel.py` itself documents, in `cpu_count_physical()`'s own docstring (lines 149-158), that "psutil may return `None` on platforms without the info — e.g. restricted containers" and provides a guarded wrapper (`count if count and count > 0 else fallback`). That guard is not reused by its neighbors in the same file, nor by `misc.py`'s equivalent helper:
  - `distribute_work()` (parallel.py:133-146): `if nworkers <= 0: nworkers = psutil.cpu_count(logical=False)` (line 136) — if that's `None`, `[[] for _ in range(nworkers)]` (line 137) raises `TypeError: 'NoneType' object cannot be interpreted as an integer`.
  - `applyfunc_parallel()` (parallel.py:174-219): `n_cores = min(psutil.cpu_count(logical=logical), len(iterable))` (line 190) — `min(None, int)` raises `TypeError: '<' not supported between instances of 'int' and 'NoneType'`.
  - `get_max_affordable_workers_count()` (misc.py:190-197): `n = psutil.cpu_count(logical=False) - reservedCores` (line 194) — raises `TypeError: unsupported operand type(s) for -: 'NoneType' and 'int'`.
- **Failure scenario**: Any of these three functions called with the "auto-detect worker count" default (e.g. `distribute_work(work, nworkers=0)`, or `applyfunc_parallel(items, fn)` with `n_cores=None`) on a host/container where psutil cannot determine physical core count (documented by this same codebase as a real, not hypothetical, condition) crashes instead of falling back to a sane default.
- **Suggested fix**: Route all three through the already-existing `cpu_count_physical()` (or an equivalent `None`-guard) instead of calling `psutil.cpu_count(logical=False)` directly.

### [Medium] `ensure_dir_exists` TOCTOU race under concurrent callers — src/pyutilz/system/system/fsutils.py:38-44
- **Category**: concurrency correctness (race condition)
- **Problem**: `existed = os.path.exists(directory); if not existed: os.makedirs(directory); return existed` — no `exist_ok=True`, no `try/except FileExistsError`. The check and the create are two separate syscalls with a window between them.
- **Failure scenario**: This helper lives in a package built around `parallel.py`'s `ThreadPool`/`Pool` workers and joblib backends. If N worker processes/threads each call `ensure_dir_exists(shared_output_dir)` at startup against the same not-yet-existing directory (an extremely common pattern — every worker ensuring its shared output location exists before writing), whichever loses the race gets an uncaught `FileExistsError` (POSIX) / `WindowsError [Error 183]` (Windows) from `os.makedirs`, crashing that worker.
- **Suggested fix**: `os.makedirs(directory, exist_ok=True)` (accepting the documented perf trade-off vs. exists-then-skip) or wrap the `os.makedirs` call in `try/except FileExistsError: pass`.

### [Medium] `UtilizationMonitor.get_average_utilization()` crashes if `get_own_memory_usage()` ever returns `None` — src/pyutilz/system/hardware_monitor.py:97, 184-197; src/pyutilz/system/system/memory.py:46-49, 79-92
- **Category**: correctness / edge-case
- **Problem**: `get_own_memory_usage()`'s own docstring states it returns `None` "if psutil raises AND we have no prior successful reading to fall back on" (memory.py:47-49) — an explicitly-documented, real return value. `hardware_monitor.py`'s background thread does `self.own_ram_used.append(get_own_memory_usage())` (line 97) with no `None` handling. `get_average_utilization()` later computes `round(np.mean(self.own_ram_used), ndigits)` (line 188) over that list.
- **Failure scenario**: Reproduced directly — `np.mean([1.0, 2.0, None, 3.0])` raises `TypeError: unsupported operand type(s) for +: 'float' and 'NoneType'` (confirmed via `python -c`). If `psutil.Process(pid).memory_info()` fails even once on the very first sample of a monitoring run (before `_LAST_OWN_MEMORY_USAGE_GB` has ever been set above 0), `get_own_memory_usage()` returns `None`, that `None` lands in `self.own_ram_used`, and any later call to `get_average_utilization()` — the whole point of running `UtilizationMonitor` for a long training job, per the module's own docstring ("profiling long-running jobs") — raises instead of returning a (partial) report.
- **Suggested fix**: Filter `None` out of `self.own_ram_used` before averaging, or skip appending a `None` sample in `query_utilization()`.

### [Medium] `scheduling/prefect.py`'s `connect()` settings.ini fallback is a silent no-op — src/pyutilz/system/scheduling/prefect.py:28-43; src/pyutilz/text/strings/configfiles.py:60-64
- **Category**: correctness (CPython semantics bug)
- **Problem**: `connect(prefect_key=None)` does, when no key is passed: `strings.read_config_file(file="settings.ini", section="PREFECT", variables="prefect_key", object=locals())` (lines 31-37), then checks `if prefect_key:` (line 38). `read_config_file` mutates its `object` argument via plain dict-item assignment (`object[var] = val`, `configfiles.py:60-64`). In CPython, `locals()` inside an ordinary (non-exec/non-module) function scope returns a **disconnected snapshot dict** — mutating that dict never writes back to the real fast-local variable slot.
- **Failure scenario**: Verified directly by reproduction (`python -c` snippet writing into `locals()` inside a function and observing the real parameter unchanged afterward). Consequently, `connect()` can **never** actually pick up `prefect_key` from `settings.ini` no matter what the file contains — `prefect_key` stays `None` inside the function after the call, `client` is set to `None` unconditionally (line 42), and `graphql.connect(client)` is never invoked with a real client unless the **caller** passes `prefect_key` explicitly, defeating the documented purpose of the settings.ini fallback path entirely.
- **Suggested fix**: Have `read_config_file` return the resolved value(s) instead of mutating a caller-supplied dict, or read the value from `read_config_file`'s written dict and explicitly reassign the real local: `cfg = {}; strings.read_config_file(..., object=cfg); prefect_key = cfg.get("prefect_key")`.

### [Medium] `wait_for_absense_of_tasks`'s `labels` parameter is accepted but ignored — src/pyutilz/system/scheduling/prefect.py:112-133
- **Category**: correctness (API contract violation)
- **Problem**: The function signature accepts `labels: Optional[set] = None` (line 115) and documents it via the surrounding module's "coordinating concurrent ML flows" purpose, but the body hardcodes `allof_labels=set(["ml", "gpu"])` at the one call site that matters (line 125) — the `labels` parameter is never read or used anywhere in the function body.
- **Failure scenario**: A caller does `wait_for_absense_of_tasks(labels={"train", "critical"})` expecting the wait to block on flows carrying those labels; instead it always waits on flows labeled `"ml"` **and** `"gpu"`, silently ignoring the caller's actual intent — it may return `True` (proceed) while flows the caller actually cared about are still running, or block on unrelated `ml`+`gpu` flows the caller didn't intend to wait for.
- **Suggested fix**: Pass `allof_labels=labels or {"ml", "gpu"}` through to `get_running_flows`.

### [Medium] `subprocess.check_output("nvcc --version")` is broken on Linux/macOS — src/pyutilz/system/system/sysinfo.py:246
- **Category**: correctness / cross-platform
- **Problem**: `cuda_version = re.findall(", V(.+)\r\n", subprocess.check_output("nvcc --version").decode())` passes a single **string containing a space** as `args` with `shell=False` (the default). Per documented Python `subprocess` semantics, on POSIX a string `args` (without `shell=True`) is interpreted as the literal path/name of the executable to run, with **no** whitespace splitting into argv — there is no executable literally named `nvcc --version`, so this always raises `FileNotFoundError` on Linux/macOS regardless of whether CUDA is installed and `nvcc` is genuinely on `PATH`. (It only appears to work on Windows, where `CreateProcess` accepts a whole command-line string and does its own parsing — which is presumably why this shipped unnoticed; every other subprocess call in this same file correctly uses list-style argv, e.g. `["cat", "/etc/machine-id"]` at line 138, `["getprop", "ril.serialnumber"]` at line 146.) The failure is silently swallowed by the surrounding `except Exception: cuda_version = [""]` (lines 247-248), and even if the process invocation somehow succeeded, the regex's hardcoded `\r\n` line ending would never match real `nvcc --version` output on Linux/macOS (which uses bare `\n`) — a second, independent reason this never populates on non-Windows.
- **Failure scenario**: On a Linux GPU host with a fully working CUDA toolkit, `get_system_info()` never sets `info["cuda_version"]` even though `numba.cuda.is_available()` is `True` and `nvcc` is on PATH — silently incomplete system-info collection with no visible error (caught by a bare `except`).
- **Suggested fix**: `subprocess.check_output(["nvcc", "--version"])` and match on a line-ending-agnostic pattern (e.g. `r", V(\S+)"`).

### [Medium] `get_locale_settings()` mutates the process-wide C locale with zero synchronization — src/pyutilz/system/system/misc.py:400-415
- **Category**: concurrency correctness
- **Problem**: `locale.setlocale(locale.LC_ALL, locale_name)` (line 410) changes process-global C-library locale state — it is not thread-local, and the CPython `locale` module is explicitly documented upstream as not thread-safe for concurrent `setlocale` calls from multiple threads. This function takes no lock and is exposed from a package whose sibling module (`parallel.py`) exists specifically to spin up thread pools.
- **Failure scenario**: Two threads (e.g. two `ThreadPool` workers from `applyfunc_parallel(..., use_threads=True)`, or any two callers in a multi-threaded app) call `get_locale_settings(locale_name="en_US.utf8")` and `get_locale_settings(locale_name="de_DE.utf8")` concurrently. Because the underlying C locale is one shared, mutable, process-wide resource, one thread's `setlocale` call can interleave with the other's read of `locale.localeconv()`, and — more broadly — **any other code in the process doing locale-sensitive formatting/parsing at that moment** (number formatting, `strftime`, decimal-point interpretation in numeric parsing) observes a temporarily-wrong locale, with no error raised.
- **Suggested fix**: Guard with a module-level lock, and/or document loudly that this function has process-wide side effects unsafe to call from worker threads.

### [Low-Medium] `count_app_instances()`: unprotected `proc.name()` amid live process-table churn — src/pyutilz/system/system/misc.py:200-223
- **Category**: robustness / race condition
- **Problem**: The `cmdline` filter path (lines 215-221) is correctly wrapped in `try/except Exception` to tolerate `psutil.NoSuchProcess`/`AccessDenied` for a process that exits mid-iteration, but the `processname` filter path calls `proc.name()` (line 213) with **no** such protection, even though it runs first, unconditionally, whenever `processname` is given.
- **Failure scenario**: `count_app_instances(processname="python.exe")` iterating `psutil.process_iter()` while a short-lived process (e.g. a build step, a spawned subprocess) exits between enumeration and the `.name()` call — a realistic race under any real process churn — raises `psutil.NoSuchProcess`, uncaught, crashing the whole count instead of just skipping that one process.
- **Suggested fix**: Wrap `proc.name()` the same way `proc.cmdline()` already is.

### [Low] `TomlLiveConfig.get()`: bool/int aliasing skips the intended type cast — src/pyutilz/system/config.py:156
- **Category**: correctness / edge-case
- **Problem**: `if type_ is not type(val) and not isinstance(val, type_):` — with the default `type_=int`, a TOML boolean value (`val = True`/`False`) satisfies `isinstance(val, int)` (bool subclasses int in Python), so the cast is skipped and the method returns a `bool`, not an `int`.
- **Failure scenario**: `cfg.get("section", "enabled", default=0, type_=int)` against a TOML key `enabled = true` returns `True` (a `bool`), not `1` (an `int`). Numeric-equality checks (`== 1`) still happen to work due to Python's bool/int equivalence, but `f"{cfg.get(...)}"` yields `"True"` instead of `"1"`, and any strict `type(x) is int` check downstream fails.
- **Suggested fix**: Exclude `bool` explicitly, e.g. `if (type_ is not type(val) or type_ is bool) and not isinstance(val, type_):`, or special-case `type_ is bool`.

### [Low] `registry.py`'s `retune_all()` docstring self-contradicts `discover_tuners()`'s own docstring — src/pyutilz/performance/kernel_tuning/registry.py:200-206 vs 274
- **Category**: docs accuracy (in-file self-contradiction, directly bearing on registry-state correctness reasoning)
- **Problem**: `discover_tuners()`'s docstring (lines 200-206) explicitly states "We do NOT clear the registry first: ... clearing would therefore wipe specs we can never re-register in the same process." `retune_all()`'s own "Side effects" section, 70 lines later in the same file, says "Calls discover_tuners (clears + repopulates registry)" (line 274) — the opposite claim, about the same function, in the same file.
- **Failure scenario**: Someone auditing `retune_all()`'s side effects (e.g. to reason about whether it's safe to call twice, or what state survives between calls) reads the wrong (more prominent, "Side effects:"-labeled) claim and draws an incorrect conclusion about registry state.
- **Suggested fix**: Fix the `retune_all` docstring to match the actual (and correctly documented elsewhere) non-clearing behavior.

### [Low] `get_max_singledisk_free_space_gb`: possible `ZeroDivisionError` on a zero-size "rw" partition — src/pyutilz/system/system/fsutils.py:47-73
- **Category**: edge-case
- **Problem**: `percent = du.free / du.total` (line 71) has no guard for `du.total == 0`. `psutil.disk_partitions()` with default `all=False` normally excludes pseudo-filesystems, making this unlikely in practice, but it is not impossible on unusual mount configurations.
- **Failure scenario**: A "rw" partition reporting `total=0` (some virtual/overlay filesystem edge cases) raises `ZeroDivisionError` inside a loop that otherwise degrades cleanly.
- **Suggested fix**: `if du.total: percent = du.free / du.total`.

## Things done well

- The v3 kernel-tuning cache's core local-write path genuinely delivers on the CHANGELOG's "no lost updates under concurrent writers" claim: `_atomic_write_json` (cache_class.py:153-179) uses a uniquely-named temp file per attempt + `os.replace`, with a bounded retry specifically documented against a real Windows AV/share-delete failure mode; `_persist_kernel` never does read-modify-write, so concurrent writers of the same or different kernels can never clobber each other (verified by reading the whole write/read path).
- `_try_create_marker`'s staged-temp-file + `os.link()` publish (cache_class.py:836-893) is careful, battle-tested engineering: it closes a real observed race ("expected one sweep, got 2" on a 2-core CI box) where a marker could be observed in a content-less state between `O_CREAT|O_EXCL` and the payload write.
- `_pid_alive()` (cache_base.py:269-306) correctly handles cross-platform PID liveness, including a documented, non-obvious Windows `ctypes` bug fix (`use_last_error=True` needed because the shared `ctypes.windll.kernel32` handle doesn't capture the Win32 thread-local last-error).
- `TomlLiveConfig`'s hot-reload path is genuinely thread-safe on inspection: `_reload()` always replaces `self._data` as a whole new dict object (never mutated in place) under `self._lock`, and `get()`'s lock-free read is safe because of GIL-atomic reference reassignment — the "thread-safe" docstring claim holds up.
- `monitoring.py`'s `timeout_wrapper` has an unusually well-reasoned design note explaining *why* it deliberately does not reuse the shared bounded `_TIMEOUT_EXECUTOR` pool (a genuine incident: permanently-stuck workers from real timeouts silently exhausting pool capacity and causing spurious timeouts for unrelated calls) — a good example of documenting a rejected simpler design.
- `memory.py`'s `get_own_memory_usage()` docstring for the Windows `rss` vs `private` distinction is excellent: concrete empirical before/after numbers, a dated incident, and a clear explanation of *why* the naive approach was wrong.
- `_gc_kernel_dir` (cache_class.py:403-419) bounds per-kernel immutable-file accumulation with a simple, best-effort keep-N-newest policy — avoids unbounded disk growth without adding any locking risk.
- `gpu_dispatch.py`'s shared-memory budget table is sourced and dated against the NVIDIA CUDA Programming Guide with per-entry comments explaining the per-block-vs-per-SM distinction — good provenance for a table that's easy to get subtly wrong.

## Investigated, not an issue

- **Kernel-tuning cache: simultaneous legacy migration by two processes.** Verified the `O_EXCL` claim genuinely serializes this correctly — the *only* real gap is crash-recovery (see High finding above), not concurrent-migration correctness.
- **`functools.lru_cache`-wrapped probes** (`_select_best_gpu_cached`, `hw_fingerprint`, `compute_code_version`, `_cpu_model_slug`, `_gpu_summary_cached`): all safe under concurrent calls — CPython's `lru_cache` internally locks cache-dict access; worst case under a race is duplicate computation (wasted work), never a corrupted cache.
- **`time_backend`'s `concurrency > 1` path** (benchmark.py:199-209): each thread writes only to its own private `results[i]` list; no shared mutable state is written concurrently, so no race despite using raw `threading.Thread` rather than a pool.
- **`_HW_BUSY_CACHE` double-checked-locking pattern** (benchmark.py:73-100): safe because the cache is always replaced as a whole new tuple under the GIL — a reader outside the lock can only observe a fully-formed old or new tuple, never a torn one.
- **`gpu_dispatch.py`'s `select_best_gpu`/`dispatch_cpu_vs_gpu`**: initially suspected these were the "fastest measured variant" dispatcher named in the assignment. On reading, they are static heuristic/threshold pickers (VRAM/compute-capability/workload-size), not empirical measurement-based dispatch — the actual "measure and pick the fastest" logic lives in `kernel_tuning/benchmark.py` + `registry.py`'s `TunerSpec.choose()`, which *is* covered above (see the benchmark.py finding).
- **`pyutilz.system.__init__`'s lazy `distributed` `__getattr__`**: concurrent first-access from multiple threads is safe — CPython's import system serializes concurrent imports of the same module name via its own per-module import lock, independent of this package's code.
