# Audit: Domain — LLM (unified provider interface)

## Summary

Covered every file under `src/pyutilz/llm/` (base interface, retry helper, all seven
provider implementations, token counter, factory, exceptions, and the OpenRouter
sub-package), read in full, plus targeted verification against the installed
`anthropic` (0.105.2), `google-genai` (2.8.0) and `tiktoken` (0.13.0) packages to
confirm exception hierarchies and tokenizer mappings rather than assume them.

Overall the module is well above average for this kind of code: retry-vs-non-retry
HTTP status classification is careful and documented, the streaming retry logic
correctly avoids duplicate-token replay, per-call state is reset defensively, and
several subtle correctness traps (async-generator + tenacity, OpenAI cache-token
semantics vs. Anthropic's) are already handled or explicitly called out in comments.
The real gaps are concentrated in two places the existing tests don't reach: **retry
predicates built against the wrong exception hierarchy** (Gemini and, more narrowly,
Anthropic), and **cost-accounting arithmetic that silently drifts from what each
provider's API actually reports** (Anthropic under-reports, OpenAI/xAI over-report).
There is also one confirmed, reproducible blocking-call-in-async-function bug in the
OpenRouter provider that stalls the entire event loop on the first call.

Findings: 3 High, 6 Medium, 5 Low (listed below, most severe first; counted after, not before).

## Findings

### [HIGH] Gemini retry predicate references the wrong exception hierarchy — never retries real Gemini API errors — src/pyutilz/llm/gemini_provider.py:34-45,150-156

- **Category**: correctness (retry logic)
- **Problem**: The retry-worthy exception tuple is built from `google.api_core.exceptions` (`ResourceExhausted`, `ServiceUnavailable`, `InternalServerError`, `TooManyRequests`), but the actual installed `google-genai` SDK (the only Gemini client this module imports — `from google import genai`) raises its own `google.genai.errors.ClientError` / `ServerError` (both inherit directly from `Exception`, verified via `google.genai.errors.APIError.__mro__ == (APIError, Exception, BaseException, object)`). `google-genai`'s own dependency list (`pip show google-genai`) does not even include `google-api-core` — it is a separate library used by older, gRPC-based Google client SDKs. The `except ImportError: _GENAI_RETRYABLE_EXCEPTIONS = ()` fallback suggests the author assumed the import would fail when `google-api-core` isn't installed, but in this environment it succeeds (pulled in transitively by other packages) while still being the wrong exception family — so the bug is silent either way.
- **Failure scenario**: A Gemini call is rate-limited (HTTP 429) or the service is briefly down (5xx). `google-genai` raises `google.genai.errors.ClientError`/`ServerError`. The `@retry(retry=retry_if_exception_type((ConnectionError, TimeoutError, OSError, *_GENAI_RETRYABLE_EXCEPTIONS)), **INFINITE_RETRY_KWARGS)` decorator's predicate does not match any of these classes, so tenacity does not retry — the exception propagates immediately out of `generate()`, aborting the caller's pipeline on the very first transient rate-limit hit instead of backing off and retrying as the module's own docstring promises ("so that long-running pipeline batches survive temporary outages, rate-limit windows... without operator intervention").
- **Suggested fix**: Retry on `google.genai.errors.ClientError` (filtered to 429, since 4xx like 400/401/403 shouldn't be retried) and `google.genai.errors.ServerError` (5xx), e.g. `retry_if_exception(lambda e: isinstance(e, genai.errors.ServerError) or (isinstance(e, genai.errors.ClientError) and getattr(e, "code", None) == 429))`, and drop the `google.api_core.exceptions` import entirely (or keep it only as a defensive extra, not the primary mechanism).

### [HIGH] Anthropic retry predicate misses OverloadedError (529), ServiceUnavailableError (503), DeadlineExceededError (504) — src/pyutilz/llm/anthropic_provider.py:124-132

- **Category**: correctness (retry logic)
- **Problem**: `retry_if_exception_type((anthropic.RateLimitError, anthropic.APIConnectionError, anthropic.APITimeoutError, anthropic.InternalServerError))` omits three exception classes the installed SDK (`anthropic==0.105.2`, satisfying the project's `anthropic>=0.40` pin) defines as **separate, sibling subclasses of `APIStatusError`** — not subclasses of `InternalServerError`: `class OverloadedError(APIStatusError): status_code = 529`, `class ServiceUnavailableError(APIStatusError): status_code = 503`, `class DeadlineExceededError(APIStatusError): status_code = 504` (confirmed by reading `anthropic._exceptions` source directly). `InternalServerError` is the generic 5xx fallback used only when no more specific subclass matches, so it does not catch any of these three.
- **Failure scenario**: Anthropic returns HTTP 529 `overloaded_error` — explicitly documented by Anthropic as a common, always-retry-worthy condition during high API load (their own error-codes guidance: "Retry with exponential backoff. Consider using a different model... spreading requests over time"). The SDK raises `anthropic.OverloadedError`, which the retry predicate does not match, so it propagates immediately and crashes the caller's batch instead of backing off — precisely the scenario `_retry.py`'s module docstring says this machinery exists to survive.
- **Suggested fix**: Add `anthropic.OverloadedError`, `anthropic.ServiceUnavailableError`, `anthropic.DeadlineExceededError` to the retry tuple (or retry on the common base where semantically safe, e.g. `isinstance(exc, anthropic.APIStatusError) and exc.status_code in {429, 500, 503, 504, 529}` instead of enumerating leaf classes that will keep drifting as the SDK adds new ones).

### [HIGH] Blocking synchronous `httpx.get()` reachable from `OpenRouterProvider`'s async `generate()`/`generate_stream()` — stalls the whole event loop — src/pyutilz/llm/openrouter_provider/_catalogue.py:48 (chain: src/pyutilz/llm/openrouter_provider/_provider.py:243-264, src/pyutilz/llm/openai_compat.py:409-410,586-587)

- **Category**: correctness (async / event-loop blocking)
- **Problem**: `OpenAICompatibleProvider.generate()` (and `generate_stream()`) resolve `max_tokens` from `self.max_output_tokens` whenever the caller passes the documented default `max_tokens=0` ("0 = use provider max"). For `OpenRouterProvider`, `max_output_tokens` (and `context_window`) are overridden to call `_resolve_model_limits(self.model_name)` → `_fetch_models_catalogue()`, which on a cache miss does `resp = _pkg().httpx.get(_pkg()._MODELS_URL, timeout=timeout)` — a fully synchronous network call with no `await`, no `asyncio.to_thread`, no executor offload. Because there is no `await` point anywhere in this call chain, the running coroutine cannot yield back to the event loop while the HTTP request is in flight.
- **Failure scenario**: A service built on `OpenRouterProvider` inside an asyncio event loop (e.g. a FastAPI/aiohttp backend serving several concurrent users, or any pipeline running several `LLMProvider`s concurrently) calls `await provider.generate(prompt)` without an explicit `max_tokens=` — the normal/documented way to call it. On the first call per process (or any call after `clear_openrouter_caches()`), this synchronously blocks the *entire* event loop — every other in-flight coroutine, including unrelated requests from other users — for however long `https://openrouter.ai/api/v1/models` takes to respond (bounded by the hardcoded `timeout=10.0`, so up to ~10 seconds of total application freeze on a single request).
- **Suggested fix**: Route `_fetch_models_catalogue()`'s HTTP call through the already-async `self._client` (an `httpx.AsyncClient`) or wrap the sync `httpx.get()` in `await asyncio.to_thread(...)`, and make `max_output_tokens`/`context_window` either genuinely async-safe or pre-warm the catalogue cache during provider construction/first async call instead of lazily inside the hot request path.

### [MEDIUM] `AnthropicProvider.get_session_cost()` double-subtracts cache tokens — silently under-reports input cost — src/pyutilz/llm/anthropic_provider.py:301-330

- **Category**: correctness (cost accounting)
- **Problem**: Anthropic's Messages API reports `usage.input_tokens` as **already excluding** both `cache_creation_input_tokens` and `cache_read_input_tokens` (total tokens sent = `input_tokens + cache_creation_input_tokens + cache_read_input_tokens` — this is Anthropic's own documented semantics, and the file's own `count_tokens()` docstring three methods above implicitly relies on the same fact). `get_session_cost()` nonetheless computes `plain_input = max(0, total_input - self.total_cache_creation_input_tokens - self.total_cache_read_input_tokens)`, subtracting the cache totals a **second time** from a figure that never included them. The comment justifying this (line ~313: "the Anthropic `input_tokens` field already excludes cache_read tokens but not cache_creation") is factually wrong about the `cache_creation` half.
- **Failure scenario**: Verified numerically (`plain_input`/`cost` computed by hand against the code's own formula): in any session where cumulative `cache_creation_input_tokens + cache_read_input_tokens` exceeds cumulative genuinely-fresh `input_tokens` (the common case for an agentic session with a large cached system prompt and many small follow-up questions), `max(0, …)` clamps `plain_input` to `0` and the entire "fresh" tier of input cost is silently dropped from `input_cost_usd`/`total_cost_usd` — a session with 5 calls (1 cache-write of 50K tokens, 4 cache-reads of 50K tokens each, 800 total tokens of genuinely fresh input across the 5 calls) reports `$0.2475` instead of the correct `$0.2499`, and the gap grows in both relative and absolute terms as the per-call "fresh" portion of a heavily-cached session grows. This directly contradicts the "won't silently under-report cost" expectation implied by the CHANGELOG's "authoritative... billed cost" framing for the LLM subsystem.
- **Suggested fix**: `plain_input = total_input` (Anthropic's `input_tokens` is already the uncached remainder); do not subtract the cache totals again.

### [MEDIUM] OpenAI / xAI cache-hit tokens never tracked — dead pricing table, cost silently over-reported — src/pyutilz/llm/openai_compat.py:501-530 (base `_record_usage`), no override in src/pyutilz/llm/openai_provider.py or src/pyutilz/llm/xai_provider.py; contrast src/pyutilz/llm/openrouter_provider/_provider.py:397-408

- **Category**: correctness (cost accounting)
- **Problem**: `OpenAICompatibleProvider._record_usage()` (the shared base every OpenAI-compatible provider funnels through) only reads `usage.get("prompt_cache_hit_tokens", 0)` — DeepSeek's field name. OpenAI's actual Chat Completions response reports cache hits under `usage.prompt_tokens_details.cached_tokens` (a *different* key, nested one level deeper); xAI, whose API is explicitly OpenAI-compatible, uses the same nested shape. Neither `OpenAIProvider` nor `XAIProvider` overrides `_track_provider_specific_usage()` to read this field. This is not a hypothesis: `OpenRouterProvider._track_provider_specific_usage()` (lines 397-408) explicitly reads **both** `usage.prompt_tokens_details.cached_tokens` **and** the legacy `usage.prompt_cache_hit_tokens`, with a comment acknowledging the base class only tracks the legacy field — proving the gap was known to whoever wrote the OpenRouter override, but was never backported to the shared base or to `OpenAIProvider`/`XAIProvider` directly.
- **Failure scenario**: A GPT-5 pipeline that repeatedly sends a large, automatically-cached system prompt (OpenAI's server-side prompt caching activates automatically above ~1024 tokens, no opt-in) accrues `total_cache_hit_tokens == 0` forever in `OpenAIProvider`/`XAIProvider`, because the field name it's looking for is never present in the response. `get_session_cost()`'s `cache_miss = total_prompt_tokens - total_cache_hit_tokens` therefore treats every prompt token as full-price cache-miss, and the whole `_CACHE_HIT_COST` pricing table defined for both providers (openai_provider.py:49-66, xai_provider.py:61-77) is unreachable dead code. For a workload where a large majority of input tokens are cache hits, `get_session_cost()["input_cost_usd"]` over-states real spend by roughly the cache discount fraction (e.g. close to 2x for OpenAI's ~90%-off cache tier on an 80%-cached workload).
- **Suggested fix**: In the shared `_record_usage`, also check `usage.get("prompt_tokens_details", {}).get("cached_tokens")` (mirroring the logic already written for OpenRouter), or give `OpenAIProvider`/`XAIProvider` a `_track_provider_specific_usage` override that does so.

### [MEDIUM] `ClaudeCodeProvider` CLI-path timeout is never retried — src/pyutilz/llm/claude_code_provider.py:406-412 (except clause) and 609-664 (`raise subprocess.TimeoutExpired`)

- **Category**: correctness (retry logic)
- **Problem**: `generate()`'s manual retry loop catches `(ConnectionError, TimeoutError, OSError)` and retries with backoff. `_generate_cli()`'s inner `run_cli()` raises `subprocess.TimeoutExpired(proc.args, self.timeout)` on line 664 when the `claude` CLI hangs past `self.timeout` (default 2400s). `subprocess.TimeoutExpired` is a `SubprocessError` subclass — **not** a subclass of the builtin `TimeoutError`, `OSError`, or `ConnectionError` (verified: `subprocess.TimeoutExpired.__mro__ == (TimeoutExpired, SubprocessError, Exception, BaseException, object)`; `issubclass(subprocess.TimeoutExpired, TimeoutError)` is `False`). It therefore falls through to `except Exception as e: if not _is_rate_limit_error(e): raise` — a timeout message never matches the rate-limit regex, so it re-raises immediately with zero retries.
- **Failure scenario**: Using the CLI fallback (SDK not installed, or `_HAS_SDK is False`), a transient hang in the `claude` subprocess (network blip mid-generation, CLI update prompt, etc.) that exceeds `self.timeout` crashes the whole `generate()` call outright instead of being retried the way the surrounding exception-handling structure clearly intends for other transient failures.
- **Suggested fix**: Add `subprocess.TimeoutExpired` (and `subprocess.SubprocessError` generally) to the retryable-exception tuple in `generate()`'s except clause.

### [MEDIUM] `generate_batch()` leaks/orphans in-flight tasks when the consumer stops iterating early — src/pyutilz/llm/base.py:312-350 and src/pyutilz/llm/openai_compat.py:684-710

- **Category**: correctness / efficiency (async task lifecycle)
- **Problem**: Both `LLMProvider.generate_batch` and `OpenAICompatibleProvider.generate_batch` (duplicated implementations) build `tasks = [asyncio.create_task(process_request(req)) for req in requests]` — scheduling **every** request's HTTP call immediately — then `for coro in asyncio.as_completed(tasks): yield await coro`. If the caller stops consuming the async generator before all tasks complete (an early `break`, an exception in the caller's loop body, or simply not fully draining it), Python delivers `GeneratorExit` into the generator at the `yield` point, but nothing in either implementation catches that to cancel the remaining `tasks`.
- **Failure scenario**: `async for result in provider.generate_batch(thousand_requests): if result.get("error"): break` — the moment a caller breaks after seeing the first error, the other ~999 already-scheduled tasks keep running to completion in the background, invisibly continuing to make real (billable) LLM API calls with no way for the caller to observe or cancel them, and any exceptions those orphaned tasks eventually raise become silently-dropped "Task exception was never retrieved" warnings.
- **Suggested fix**: Wrap the `for coro in asyncio.as_completed(tasks): yield await coro` loop in `try/finally`, cancelling any task still in `tasks` that isn't done when the generator exits (`finally: [t.cancel() for t in tasks if not t.done()]`), or use `asyncio.TaskGroup`/bounded concurrency instead of scheduling everything up front unconditionally.

### [MEDIUM] Token counting silently falls back to the wrong tokenizer for every non-Anthropic-API, non-Gemini-API path and mislabels it "(accurate)" — src/pyutilz/llm/token_counter.py:1-37, src/pyutilz/llm/openai_compat.py:738-741, src/pyutilz/llm/claude_code_provider.py:746-749 (contrast src/pyutilz/llm/anthropic_provider.py:234-259)

- **Category**: correctness / docs (token counting accuracy)
- **Problem**: `token_counter.count_tokens()` hardcodes `tiktoken.get_encoding("cl100k_base")` and is used, without any provider-specific correction, by `OpenAICompatibleProvider.count_tokens()` (base for DeepSeek, xAI, OpenAI, OpenRouter — docstring literally says `"Count tokens using tiktoken (accurate) or len//4 fallback."`) and by `ClaudeCodeProvider.count_tokens()` (same docstring, same "(accurate)" label). This encoding is wrong for every one of these except legacy GPT-4/3.5: verified directly against the installed `tiktoken` package that `gpt-4o` and `o1` resolve to `o200k_base`, not `cl100k_base` — so even `OpenAIProvider`'s own current model lineup (`gpt-5`, `gpt-4.1`, `o1`, `o3`, `o4-mini`, all present in its own `_PRICING` table) is mis-tokenized by the "(accurate)" fallback. `AnthropicProvider.count_tokens()` (a few files over, in the same package) explicitly documents the opposite conclusion about the same encoding: "Tiktoken — the previous fallback — uses OpenAI's tokenizer, which is the WRONG tokenizer for Claude (the cl100k_base mapping diverges from Claude's BPE for >5% of typical text)" and calls Anthropic's real `messages.count_tokens` API instead. `ClaudeCodeProvider` runs the exact same Claude models but never calls that API from `count_tokens()` — it routes straight to the mislabeled tiktoken fallback.
- **Failure scenario**: Any caller using `ClaudeCodeProvider` (the **default** provider returned by `get_llm_provider()` with no arguments) or any OpenAI-compatible provider's `count_tokens()` for a pre-flight "does this fit in the context window" or "am I under budget" check gets a count that is wrong by the margin documented elsewhere in this same file (>5% typical text, "much more... for non-English input" per the same reasoning already accepted for `AnthropicProvider`), with no indication in the API surface that the number is an approximation — the docstring says the opposite.
- **Suggested fix**: For `ClaudeCodeProvider`, delegate `count_tokens()` to a real Anthropic client's `messages.count_tokens` (same approach `AnthropicProvider` already uses) since Claude Code is running actual Claude models. For the OpenAI-compatible family, at minimum stop calling the tiktoken fallback "(accurate)" in the docstring, and consider mapping known OpenAI model names to `o200k_base` vs `cl100k_base` via `tiktoken.encoding_for_model()` instead of a single hardcoded encoding.

### [MEDIUM] `LLMTruncationError` is fully specified but never raised anywhere — src/pyutilz/llm/exceptions.py:45-53

- **Category**: correctness / docs (silent dead contract)
- **Problem**: The exception class carries a precise, actionable contract in its docstring — "LLM stopped because `max_tokens` was hit mid-generation. Retryable — caller should double `max_tokens` (capped) and re-issue" — and a `finish_reason` field to support that. Every provider captures the equivalent signal (`self._last_finish_reason = response.stop_reason` in Anthropic, `choices[0].get("finish_reason", "unknown")` in the OpenAI-compat base, Gemini's `finish_reason`), but none of them ever compares it to the truncation sentinel (`"max_tokens"` / `"length"`) and raises `LLMTruncationError`. A grep across `src/pyutilz/llm/` confirms zero `raise LLMTruncationError(...)` call sites.
- **Failure scenario**: A caller building a `generate_json()`-based pipeline that specifically catches `LLMTruncationError` to auto-double `max_tokens` and retry — exactly the pattern the docstring recommends — never sees that exception fire, even when a response genuinely was cut off mid-JSON by hitting `max_tokens`; instead they get a generic `JSONParsingError` from `extract_json()` (unbalanced braces), losing the more specific, actionable signal the codebase advertises as available.
- **Suggested fix**: Either wire `LLMTruncationError` into each provider's `generate()` (raise when `finish_reason` indicates truncation), or remove/mark it clearly experimental if it isn't meant to be load-bearing yet.

### [LOW] Inconsistent missing-API-key error messages — src/pyutilz/llm/anthropic_provider.py:66, src/pyutilz/llm/gemini_provider.py:98 vs. src/pyutilz/llm/deepseek_provider.py:63, src/pyutilz/llm/xai_provider.py:107, src/pyutilz/llm/openai_provider.py:134, src/pyutilz/llm/openrouter_provider/_provider.py:106

- **Category**: edge-case / UX
- **Problem**: `DeepSeekProvider`/`XAIProvider`/`OpenAIProvider`/`OpenRouterProvider` all raise `ValueError` messages that name the exact environment variable to set (e.g. `"DeepSeek API key not provided. Set DEEPSEEK_API_KEY in .env or pass api_key="`). `AnthropicProvider` and `GeminiProvider` raise only `"Anthropic API key not provided"` / `"Gemini API key not provided"` with no hint about which env var or how to fix it.
- **Failure scenario**: A first-time user hits `ValueError: Anthropic API key not provided` with no indication that `ANTHROPIC_API_KEY` (via `.env`) or `api_key=` is the fix — they have to go read `config.py` to find the field name, unlike the other five providers which tell them directly in the exception message.
- **Suggested fix**: Match the more informative message pattern used by the other four providers.

### [LOW] `AnthropicProvider.max_output_tokens` uses a fragile substring match instead of the module's own prefix-matching helper — src/pyutilz/llm/anthropic_provider.py:98-109

- **Category**: correctness / maintainability
- **Problem**: `if self.model.startswith("claude-opus"): return 128000 if "4-6" in self.model else 32000` — unlike the class's own `_PRICING` table, which is resolved through `_longest_prefix_pricing()` (a deliberately careful longest-prefix matcher specifically written, per its own docstring, to avoid "a future `claude-opus-4-2-YYYYMMDD` inheriting `claude-opus-4-7`'s cheaper tier just because that entry iterated first"), `max_output_tokens` uses a bare substring test that has exactly the failure mode that helper was written to avoid.
- **Failure scenario**: Any current or future Opus model whose id doesn't literally contain the substring `"4-6"` (e.g. `claude-opus-4-7-...`, `claude-opus-4-8-...`, or any Opus release with a different numbering scheme) silently gets `max_output_tokens == 32000` instead of the real (larger, e.g. 128000) limit — `generate()`'s `max_tokens = min(self.max_output_tokens, 21000)` default wouldn't itself notice, but any caller reading `provider.max_output_tokens` directly to size a request gets a number that's wrong by 4x for a model this same file's `_PRICING` table already prices correctly.
- **Suggested fix**: Drive `max_output_tokens` off the same kind of per-model-family lookup table (or the same `_longest_prefix_pricing`-style resolver) used for pricing, rather than a hand-rolled substring check.

### [LOW] `_retry.py`'s documented backoff sequence doesn't match what `wait_exponential` actually produces — src/pyutilz/llm/_retry.py:47-49

- **Category**: docs
- **Problem**: The comment describes "exponential 5s → 10s → 20s → ... → 300s (5 min cap)", but `wait_exponential(multiplier=2, min=5, max=300)` (base 2 implicit) actually computes `2 * 2**(attempt-1)` clamped to `[5, 300]`, which for attempts 1-9 is `5, 5, 8, 16, 32, 64, 128, 256, 300` — the first two attempts both floor at 5s (not 5, then 10) before the doubling pattern becomes visible, and the values never actually hit round numbers like 10/20/40.
- **Failure scenario**: None functionally — this is a documentation/comment accuracy issue only; the mechanism itself is correct (increasing backoff with jitter, capped at 300s).
- **Suggested fix**: Update the comment to describe the actual sequence, or simplify it to "roughly doubling from 5s up to a 300s cap".

### [LOW] `ClaudeCodeProvider` reimplements its own retry loop instead of using the shared `_retry.py` — no jitter, and ignores `PYUTILZ_LLM_MAX_RETRIES` — src/pyutilz/llm/claude_code_provider.py:349,406-412 vs. src/pyutilz/llm/_retry.py:24-28,49

- **Category**: correctness (minor) / maintainability
- **Problem**: Every other provider in this package goes through `_retry.py`'s shared `INFINITE_RETRY_KWARGS` (backoff with `wait_random(0, 5)` jitter, retry-count bound from the `PYUTILZ_LLM_MAX_RETRIES` env var, default 50, 0 = infinite). `ClaudeCodeProvider.generate()` instead hand-rolls its own loop: `wait = min(5 * (2 ** (attempt - 1)), 300)` (no jitter term at all) and a hardcoded `max_attempts = 20` that never reads `PYUTILZ_LLM_MAX_RETRIES`.
- **Failure scenario**: (a) No jitter — if several `ClaudeCodeProvider` workers in the same process/host hit a transient connection error at the same moment (e.g. a shared network blip), they all retry in lockstep at 5s/10s/20s/... intervals, a classic thundering-herd pattern that `_retry.py`'s jitter was specifically added to avoid elsewhere. (b) Since `get_llm_provider()`'s default `provider_name` is `"claude-code"`, this is the retry behavior most callers get by default, and it silently diverges from the env-var-configurable behavior documented at the top of `_retry.py` ("All providers use retry... configurable via `PYUTILZ_LLM_MAX_RETRIES`").
- **Suggested fix**: Reuse `_retry.py`'s `RETRY_WAIT`/`MAX_RETRY_ATTEMPTS` (or at minimum add the same `random.uniform(0, 5)` jitter term and read the same env var) instead of a parallel hand-rolled implementation.

### [LOW] `GeminiProvider` offloads the SDK call to a dedicated `ThreadPoolExecutor` instead of using `google-genai`'s native async client — src/pyutilz/llm/gemini_provider.py:186-194,320-327

- **Category**: efficiency
- **Problem**: `generate()` and `count_tokens()` both do `await loop.run_in_executor(self._executor, lambda: self.client.models.generate_content(...))` / `.count_tokens(...)`, correctly avoiding blocking the event loop — but the installed `google-genai` SDK exposes a native async surface (`self.client.aio.models.generate_content`, `.aio.models.count_tokens`, confirmed present via introspection) that would avoid consuming a thread-pool worker per concurrent call and integrate more naturally with `asyncio` cancellation.
- **Failure scenario**: Under high `max_concurrent`, every in-flight Gemini call occupies one thread-pool worker (bounded by `ThreadPoolExecutor(max_workers=max_concurrent)`) for the full duration of the HTTP round-trip, which is strictly less efficient than native async I/O, though not incorrect.
- **Suggested fix**: Switch to `self.client.aio.models.generate_content(...)` / `.aio.models.count_tokens(...)` and drop the dedicated executor (keeping `_close()`'s executor-shutdown logic only if still needed elsewhere).

## Things done well

- `generate_stream()`'s retry logic (openai_compat.py) explicitly tracks whether any content has already been yielded (`emitted_any`) and refuses to retry once it has, specifically to avoid duplicating already-delivered tokens on a resumed stream — a correct, non-obvious answer to exactly the kind of non-idempotency question this audit was asked to look for.
- The same file's docstring explicitly calls out that `@retry` is a no-op on async-generator functions (a genuine, easy-to-miss tenacity gotcha) and documents why the manual retry loop exists instead — this is exactly the sort of subtlety that's easy to get wrong silently, and it's handled correctly and knowingly.
- `_longest_prefix_pricing()` (base.py) is a genuinely careful piece of engineering: it explicitly reasons about dict-iteration-order nondeterminism and picks the longest matching prefix (with a documented two-pass fallback for date-suffixed vs. date-less model ids) rather than the naive "first `startswith` match" that would silently mis-price a new model sharing a prefix with a cheaper one.
- `GeminiProvider.generate()`/`count_tokens()` correctly offload the synchronous SDK calls via `loop.run_in_executor` — a useful positive contrast to the OpenRouter catalogue-fetch bug found above, showing the pattern is understood and applied correctly elsewhere in the same package.
- Streaming connection cleanup on cancellation/mid-stream exception is handled correctly: the SSE `yield` sits inside `async with self._client.stream(...)`, so both an abandoned async generator (`GeneratorExit`) and a mid-stream exception properly close the underlying httpx response via ordinary `async with` semantics — no leaked connections.
- `_health.py`'s TTL-cache write logic re-takes `time.monotonic()` *after* the (potentially multi-second) HTTP fan-out completes and checks for a newer concurrent write before overwriting, specifically to avoid a subtly wrong "age every fresh entry by the fetch duration" bug — good attention to a race that's easy to miss.
- Refusal detection (`is_llm_refusal`) is deliberately conservative (phrase-anchored regexes, not loose keyword matching) with an explicit comment about why: false positives would silently degrade a valid answer into a spurious refusal.
- `_reset_per_call_state()` (OpenRouter) is called at the start of every `generate()` specifically so a failed/partial call can't leave stale `last_*` metadata from the previous successful call — avoids a realistic "wrong call's data" bug class.
- DeepSeek's HTTP 402 handling (retry indefinitely with a clear warning log) is a deliberate, documented design choice for unattended long-running batches, not an oversight — consistent with the module's stated purpose.

## Investigated, not an issue

- **Tenacity `@retry` on an async generator** — confirmed this is explicitly documented and worked around (see "Things done well"); not a hidden bug.
- **Streaming mid-stream exception leaving a connection unclosed** — traced the `async with self._client.stream(...)` / `yield` structure; confirmed both cancellation and mid-stream failure correctly trigger httpx's cleanup.
- **DeepSeek HTTP 402 retried "indefinitely until balance is restored"** — initially looked like a non-idempotent-retry-on-a-permanent-error bug, but the request never partially succeeds (the server rejects it outright for insufficient balance, no partial billing), and the behavior is explicitly intentional per both the inline comment and `_retry.py`'s module docstring ("survive... billing pauses without operator intervention"). Not a bug.
- **OpenAI-compat's `cache_miss = total_prompt_tokens - total_cache_hit_tokens` arithmetic** — checked this subtraction in isolation against OpenAI/DeepSeek's actual API semantics (where `cached_tokens`/`prompt_cache_hit_tokens` *is* a subset of `prompt_tokens`, unlike Anthropic). The subtraction itself is correct; the bug found above is entirely in the upstream field-name resolution, not this formula.
- **Anthropic's estimated "thinking token" count (`chars // 4`) being double-counted in cost** — confirmed `last_thinking_tokens` is purely informational (explicitly flagged via `last_thinking_tokens_estimated`) and is never separately added into `output_cost_usd`, which is computed from `total_output_tokens * out_rate` — and Anthropic's `usage.output_tokens` already includes thinking-token spend, so there's no double count.
- **Missing-await patterns beyond what the meta-test scanner catches** — grepped every `self.generate(`, `self._generate_sdk(`, `self._generate_cli(`, `asyncio.sleep(`, `run_in_executor(` call site across the package; all are properly awaited.
- **Live budget test (`tests/test_llm_live.py::test_anthropic_minimal_call`) as a safety net for the Anthropic cost bug** — checked: it calls `p.estimate_cost(...)` (a separate, simpler code path with no cache-token subtraction) rather than `p.get_session_cost()`, so it would not catch the bug found above even when run against a live key.
