# Packaging & Dependency Management Audit — pyutilz

## Summary

Covered `pyproject.toml` in full plus a repo-wide AST-based sweep of every top-level `import`/`from` statement in `src/pyutilz/**/*.py` (120 files), cross-referenced against `[project.optional-dependencies]`, the documented undeclared-import exceptions, `tests/test_meta/test_version_consistency.py`, `tests/test_meta/test_optional_deps_isolation.py`, and `.github/workflows/ci.yml`. The "zero hard dependencies" claim (`dependencies = []`) does **not** actually hold end-to-end: the most foundational shared helper module (`core/pythonlib.py`, imported by 12 other files across nearly every domain) unconditionally imports `numba`/`joblib`/`portalocker`, and at least eight other modules unconditionally import a package that the single extras group matching their own domain does not declare (pandas, numpy, or polars showing up repeatedly as the culprit). All of these are invisible to CI, because CI always installs a large bundle of extras together and never tests a single extras group in isolation, and the dedicated masked-import meta-test's own dependency-group map omits the `system`/`gpu` groups entirely and never imports the actual leaf submodules where the bugs live. A separate, unconditional `import tomllib` breaks `pyutilz.system.config` outright on the declared Python 3.8/3.9/3.10 floor. Findings: 1 Critical, 9 High, 4 Medium, 1 Low.

## Findings

### [CRITICAL] `core/pythonlib.py` unconditionally imports numba/joblib/portalocker — contradicts "zero hard dependencies" for the library's most-shared helper module — src/pyutilz/core/pythonlib.py:31,37,869
- **Category**: packaging / dependency-management
- **Problem**: `src/pyutilz/core/pythonlib.py` has three unconditional, unguarded, module-level third-party imports: `from numba import njit` (line 31), `import joblib` (line 37), and `import portalocker` (line 869, still at module scope — confirmed by reading the surrounding code, it sits directly before `open_safe_shelve()` with no indentation). None of these three packages appears in pyproject.toml's undeclared-import exception list (which only documents database/__init__.py+redislib.py, dev/dashlib.py, system/scheduling/prefect.py, system/parallel.py's `set_tf_gpu`, text/strings/webtext.py, core/image.py+core/matrix.py). `pyutilz.core.pythonlib` (or its alias `pyutilz.pythonlib`) is imported, directly or via `from pyutilz.core.pythonlib import X` / `from pyutilz.pythonlib import X`, by at least 12 other files spanning nearly every domain: `system/system/probing.py`, `database/db/__init__.py`, `dev/logginglib.py`, `text/strings/configfiles.py`, `data/polarslib.py`, `core/filemaker.py`, `performance/kernel_tuning/cache/__init__.py`, `system/hardware_monitor.py`, `data/pandaslib/_common.py`, `system/gpu_dispatch.py`, `text/strings/__init__.py`. There is no `core` extras group at all; `numba`/`joblib`/`portalocker` are declared only under the unrelated-by-name `system` extra (pyproject.toml:139,141,142).
- **Failure scenario**: A user runs `pip install pyutilz` (base install, matching the package description's own "zero hard dependencies, install only what you need" pitch, pyproject.toml:8) or `pip install pyutilz[polars]` / `pyutilz[database]` / `pyutilz[nlp]`, then does anything that reaches `pyutilz.core.pythonlib` — e.g. `from pyutilz.data.polarslib import ...` (which itself does `from pyutilz.pythonlib import is_cuda_available, check_cpu_flag` at polarslib.py:29) or `from pyutilz.database.db import ...` (db/__init__.py:30 imports `weekofmonth, datetime_to_utc_timestamp, lookup_in_stack` from pythonlib). Result: `ModuleNotFoundError: No module named 'numba'`, even though the user never asked for anything numba-related and has no natural reason to guess they need the `system` extra to use a database or polars helper.
- **Suggested fix**: Either (a) move `njit`/`joblib`/`portalocker` usage in pythonlib.py behind try/except with graceful no-op fallbacks (the codebase already has this exact pattern in `stats/normality.py:38-55` for numba — a ready-made template), or (b) split the numba/joblib/portalocker-dependent helpers out of `core/pythonlib.py` into a separate module that callers only reach when they actually need those features, keeping the base helpers (string/type utilities etc.) numba-free.

### [HIGH] Guard meant to catch exactly this class of bug has a structural blind spot — tests/test_meta/test_optional_deps_isolation.py:38-46,159-178
- **Category**: testing / enforcement-gap
- **Problem**: `test_optional_deps_isolation.py` (PT-8) is the meta-test explicitly designed to prevent unconditional optional-dependency imports (its own docstring: "The promise to users: `pip install pyutilz` always works... This test enforces that promise"). Its `_OPTIONAL_DEP_GROUPS` dict (lines 38-46) enumerates only `pandas`, `polars`, `database`, `web`, `cloud`, `nlp`, `llm` — it omits the `system` extra (numba, joblib, portalocker, psutil, tqdm, pympler, Pillow, scipy, py-cpuinfo, GPUtil, xmltodict) and the `gpu` extra (cupy) entirely, so those packages are never masked/simulated-absent by any scenario. Separately, `test_safe_subpackages_import_with_all_optional_deps_masked` (lines 159-178) only imports the bare packages `pyutilz`, `pyutilz.core`, `pyutilz.text` (line 51) — never the leaf submodules (`pyutilz.core.pythonlib`, `pyutilz.core.matrix`, `pyutilz.text.strings`, etc.) where the actual eager imports live. `pyutilz/core/__init__.py` and `pyutilz/text/__init__.py` are both declaration-only (just `__all__`, no actual submodule imports — confirmed by reading both files in full), so the "always importable" test trivially passes regardless of what pythonlib.py/matrix.py/strings/__init__.py actually do.
- **Failure scenario**: This is precisely why the Critical finding above (and several High findings below) shipped undetected: `from numba import njit` sits unconditionally in `core/pythonlib.py`, but no test scenario ever (a) masks numba, or (b) actually imports `pyutilz.core.pythonlib`. Every one of this report's High-severity single-extra-breaks-alone findings has the same blind spot for the same reason.
- **Suggested fix**: Add `"system": ["numba", "joblib", "portalocker", "psutil", "tqdm", "pympler", "PIL", "scipy", "cpuinfo", "GPUtil", "xmltodict"]` and `"gpu": ["cupy"]` to `_OPTIONAL_DEP_GROUPS`, and extend `_ALWAYS_IMPORTABLE_SUBPACKAGES` (or add a new parametrized case) to actually import representative leaf submodules per top-level package (e.g. `pyutilz.core.pythonlib`, `pyutilz.core.matrix`, `pyutilz.data.polarslib`, `pyutilz.data.pandaslib`, `pyutilz.text.strings`, `pyutilz.database.db`, `pyutilz.web.web`), each masked against every OTHER group's deps to simulate "I installed extra X but not Y."

### [HIGH] `import tomllib` is unconditional — breaks the declared Python 3.8/3.9/3.10 support floor — src/pyutilz/system/config.py:29
- **Category**: correctness / Python-version-floor violation
- **Problem**: `system/config.py` does `import tomllib` unconditionally at module level (line 29), with no `try/except`, no `sys.version_info` guard, and no fallback to a third-party `tomli` backport. `tomllib` is stdlib **only since Python 3.11** (PEP 680). `pyproject.toml` declares `requires-python = ">=3.8"` and classifiers list Python 3.8, 3.9, 3.10 as explicitly supported (pyproject.toml:27-29). The project is clearly aware of this exact constraint elsewhere: `tests/test_meta/test_version_consistency.py:34-36` explicitly says it "Avoids pulling in `tomllib` (Python 3.11+) to keep this test runnable on 3.8/3.9 too" — so the codebase has the right knowledge, just not applied consistently here. No test anywhere imports `TomlLiveConfig`/`pyutilz.system.config` (grepped `tests/` — only a `code_audit` JSON baseline file matches, not a functional test), so this is currently undetected on every CI leg, 3.8 through 3.14.
- **Failure scenario**: A user on Python 3.8, 3.9, or 3.10 (all declared-supported) does `from pyutilz.system.config import TomlLiveConfig` → `ModuleNotFoundError: No module named 'tomllib'`, immediately, regardless of any extras installed (this isn't even a `pip install` gap — no extra could ever supply a stdlib module).
- **Suggested fix**: `try: import tomllib \n except ImportError: import tomli as tomllib` (with `tomli` declared as a conditional dependency for `python_version < '3.11'`), matching the pattern the project already uses correctly for numba in `stats/normality.py`.

### [HIGH] `system/parallel.py` unconditionally imports pandas — `pyutilz[system]` alone cannot import it — src/pyutilz/system/parallel.py:25
- **Category**: packaging / dependency-management
- **Problem**: `import pandas as pd` at line 25 is unconditional, module-level. The `system` extra (pyproject.toml:136-152: psutil, tqdm, numba, pympler, joblib, portalocker, Pillow, scipy, py-cpuinfo, GPUtil, xmltodict) does not declare pandas, and none of those packages transitively require pandas. Real usage confirmed: `applyfunc_parallel()` (line 174-217) has `return_dataframe: bool = True` as its default (line 178) and calls `pd.concat(res, ignore_index=True)` at line 217 on that default path.
- **Failure scenario**: `pip install pyutilz[system]` then `from pyutilz.system.parallel import applyfunc_parallel` → `ModuleNotFoundError: No module named 'pandas'` at the `import pandas as pd` line, before the function is even called. (Works fine under `pyutilz[all]` or CI's bundled install, which is exactly why it's gone unnoticed — see the CI finding below.)
- **Suggested fix**: Add `pandas` to the `system` extra's declared deps, or make the import lazy/conditional inside `applyfunc_parallel` (only needed when `return_dataframe=True`).

### [HIGH] `database/db/__init__.py` unconditionally imports pandas but only `dateutil`/`redis` are documented as needing declaration — src/pyutilz/database/db/__init__.py:23
- **Category**: packaging / dependency-management
- **Problem**: `import pandas as pd` at line 23 is unconditional. The `database` extra (pyproject.toml:60-71) has an inline comment specifically calling out `database/__init__.py`'s undeclared `dateutil.relativedelta` and `redislib.py`'s undeclared `redis` as the reasons `python-dateutil`/`redis` were added — but the SAME file (`database/db/__init__.py`) also unconditionally imports pandas, and pandas is nowhere in the `database` extra's dependency list. ### [HIGH] `data/pandaslib` unconditionally requires polars too — `pyutilz[pandas]` alone cannot import it — src/pyutilz/data/pandaslib/_common.py:34
- **Category**: packaging / dependency-management
- **Problem**: `_common.py` (the shared-imports hub for the `pandaslib` subpackage, unconditionally imported by `data/pandaslib/__init__.py:15-49`) has `import pandas as pd, polars as pl` at line 34 — both unconditional. The `pandas` extra (pyproject.toml:47-51: `pandas>=1.5`, `numpy>=1.20`, `pyarrow>=10.0`) does not declare polars.
- **Failure scenario**: `pip install pyutilz[pandas]` (the extra whose name exactly matches this subpackage's domain) then `from pyutilz.data.pandaslib import ...` (or the `pyutilz.pandaslib` alias) → `ModuleNotFoundError: No module named 'polars'`. Only `pyutilz[dataframes]` (pandas+polars combined) or `pyutilz[pandas,polars]` actually works — not obvious from the extra's own name.
- **Suggested fix**: Either make `pandaslib`'s polars import lazy (guard the polars-specific helpers), or document/rename so the `pandas` extra genuinely means "pandas only" and split out the cross-library helpers into a module reachable only via `dataframes`.

### [HIGH] `data/polarslib.py` unconditionally imports numpy — `pyutilz[polars]` alone cannot import it — src/pyutilz/data/polarslib.py:23
- **Category**: packaging / dependency-management
- **Problem**: `import numpy as np` at line 23 is unconditional. The `polars` extra (pyproject.toml:52: `polars>=0.18`) does not declare numpy. Verified via the locally-installed polars 1.41.2 package's own declared `Requires-Dist` metadata (`importlib.metadata.distribution('polars').requires`) that modern polars treats numpy as **optional** (`numpy>=1.16.0; extra == "numpy"`), not a hard dependency — so `pip install pyutilz[polars]` genuinely will not pull in numpy transitively.
- **Failure scenario**: `pip install pyutilz[polars]` then `from pyutilz.data.polarslib import ...` (or `pyutilz.polarslib` alias) → `ModuleNotFoundError: No module named 'numpy'`.
- **Suggested fix**: Add `numpy` to the `polars` extra's dependency list (cheapest fix, matches how `pandas` extra already declares it explicitly rather than relying on transitive luck).

### [HIGH] `pyutilz.text.strings` package unconditionally requires pandas+numpy, but no extras group covers the "text" domain — src/pyutilz/text/strings/__init__.py:21 (+ webtext.py:11, basics.py:12, textentropy.py:14)
- **Category**: packaging / dependency-management
- **Problem**: `text/strings/__init__.py:21` does `import pandas as pd, numpy as np` unconditionally — and this `__init__.py` is what every `from pyutilz.text.strings import X` / `pyutilz.strings` / `pyutilz.string` alias goes through. There is **no** `text` or `strings` extras group in `pyproject.toml` at all. The semantically closest group, `nlp` (pyproject.toml:110-122: spacy, nltk, jellyfish, tiktoken, beautifulsoup4, inflect, emoji-data-python), does not declare pandas or numpy either. Independently, `webtext.py:11` also has its own unconditional `import pandas as pd` (not mentioned in the pyproject.toml comment documenting webtext.py's exceptions, which only lists bs4/inflect/emoji-data-python), and `basics.py:12`/`textentropy.py:14` add further unconditional pandas/numpy imports.
- **Failure scenario**: `pip install pyutilz[nlp]` (the only extras group whose domain matches "text utilities") then `from pyutilz.text.strings import find_between` (or any other string helper with zero dataframe involvement) → `ModuleNotFoundError: No module named 'pandas'`. The only way to fix it is installing the unrelated-by-name `pandas` extra as well.
- **Suggested fix**: Either declare `pandas`+`numpy` in the `nlp` extra, or (better, since most of `text.strings` genuinely has nothing to do with dataframes) make the pandas/numpy usage lazy/scoped to only the specific helper functions that need them, so plain string utilities stay dependency-free.

### [HIGH] `web/web.py` unconditionally imports joblib — `pyutilz[web]` alone cannot import it, and CI never tests `[web]` at all — src/pyutilz/web/web.py:27
- **Category**: packaging / dependency-management
- **Problem**: `from joblib import hash as joblib_hash` at line 27 is unconditional. The `web` extra (pyproject.toml:74-81: selenium, undetected-chromedriver, requests, grequests, fake-useragent, curl-cffi) does not declare joblib, and none of those packages pull it in transitively.
- **Failure scenario**: `pip install pyutilz[web]` then `from pyutilz.web.web import ...` (or `pyutilz.web` package init, which per `test_optional_deps_isolation.py:105-119`'s own history eagerly imports all of `browser, graphql, proxy, web`) → `ModuleNotFoundError: No module named 'joblib'`. This is compounded by the fact that `.github/workflows/ci.yml` (see the CI finding below) never installs `[web]` at all on any matrix leg, so this would never surface as a CI failure even after the extras dependency list is otherwise correct.
- **Suggested fix**: Add `joblib` to the `web` extra (cheap — it's a small, pure-Python dependency), or replace `joblib.hash` with `hashlib`-based hashing to drop the dependency entirely.

### [MEDIUM] CI never installs a single extras group in isolation, and never installs `[web]`/`[dash]`/`[prefect]`/`[tensorflow]` at all — .github/workflows/ci.yml:59-66
- **Category**: OSS-practice / CI-coverage gap
- **Problem**: Every matrix leg's "Install dependencies" step (lines 59-66) installs a large bundle together — `pandas,polars,database,system,cloud,dev` (3.8), `pandas,polars,database,system,llm,cloud,dev` (3.9), or `pandas,polars,database,system,llm,nlp,cloud,dev` (3.10+) — never a single extras group by itself. `web`, `dash`, `prefect`, `tensorflow`, and `gpu` are never installed by CI at all (not even bundled). This is the direct, root-cause reason every High-severity finding above (parallel.py, memory.py, db/__init__.py, pandaslib/_common.py, polarslib.py, text/strings/__init__.py, web/web.py) has shipped undetected: the bundle always happens to include whichever extra was "secretly" also needed.
- **Failure scenario**: Any future PR that adds a new unconditional third-party import to a domain module will pass CI as long as the import belongs to one of the always-co-installed extras — the exact failure mode already realized by the 6+ findings above.
- **Suggested fix**: Add matrix legs (or a separate job) that install each documented extras group by itself (`pyutilz[system]`, `pyutilz[pandas]`, `pyutilz[polars]`, `pyutilz[database]`, `pyutilz[web]`, `pyutilz[nlp]`, `pyutilz[llm]`) and run at minimum a smoke-import of every public submodule under that extra's nominal domain — this is a more direct, black-box-real fix than only patching the `test_optional_deps_isolation.py` mask lists (finding above), and would catch both classes of bug.

### [MEDIUM] numpy has no standalone extras group — several modules rely on transitive luck or have no matching extra at all — src/pyutilz/dev/benchmarking.py:16, src/pyutilz/stats/normality.py:36 (+ core/matrix.py:22, data/numbalib.py:18-19, data/numpylib.py:22, system/hardware_monitor.py:18)
- **Category**: packaging / architecture
- **Problem**: Nine files unconditionally `import numpy`, but `numpy` is declared explicitly in exactly one extras group (`pandas`, pyproject.toml:49). For `core/matrix.py`, `data/numbalib.py`, `data/numpylib.py`, and `system/hardware_monitor.py`, numpy happens to work today only because `scipy`/`numba` (both in the `system` extra) hard-depend on numpy themselves — a transitive dependency the codebase doesn't control staying stable. For `dev/benchmarking.py` and `stats/normality.py`, there is genuinely **no** extras group matching their own domain: `dev` extra is pytest/black/ruff/mypy/bandit tooling (no numpy), and there is no `stats` extra at all — despite `pyproject.toml`'s own strict-mode-mypy comment (lines 424-431) treating `pyutilz.stats` as a cohesive, stable, already-verified subpackage.
- **Failure scenario**: `pip install pyutilz[dev]` (a real extras group, reasonable for a contributor) then `from pyutilz.dev.benchmarking import synchronize_gpu_if_available` → `ModuleNotFoundError: No module named 'numpy'`, because `dev` extra installs no scientific packages at all.
- **Suggested fix**: Either add a lightweight standalone `numpy` extra, add numpy directly to `system`/`dev` as appropriate, or add numpy to a new `stats` extras group and to `dev`.

### [MEDIUM] Version-consistency meta-test's pyproject.toml parser is single-quote-fragile and can silently become a no-op — tests/test_meta/test_version_consistency.py:30-51,64
- **Category**: testing / enforcement-gap
- **Problem**: `_read_pyproject_version()` extracts the version via `re.match(r'version\s*=\s*"([^"]+)"', stripped)` (line 48) — matches only double-quoted TOML strings. If the `[project]` version line is ever reformatted to single quotes (`version = '1.0.0'`, valid TOML) or otherwise fails to match, the function silently returns `None` and the value is simply never added to `sources` (lines 60-62) — no error, no warning. Critically, `pyutilz.__version__` and `pyutilz.version.__version__` (the other two "sources") are not actually independent: `src/pyutilz/__init__.py:3` does `from .version import __version__`, so they are always the exact same object/value. This means `assert len(sources) >= 2` (line 64) is trivially satisfied by those two guaranteed-identical entries alone, and `len(distinct) > 1` (line 66) will always be `False` in that degraded state — the test keeps passing with zero actual protection against the one failure mode its docstring says it exists to catch ("Catches the failure mode where `version.py` is bumped but the `pyproject.toml` value isn't", lines 9-11).
- **Failure scenario**: A future contributor (or an automated formatter) changes `pyproject.toml:7` from `version = "1.0.0"` to `version = '1.0.1'` while `version.py` still says `"1.0.0"` — the test passes silently, and a release could ship with `pip install pyutilz==1.0.1` while `pyutilz.__version__` reports `"1.0.0"`, exactly the drift scenario the test's own docstring describes as the thing it prevents.
- **Suggested fix**: Match both quote styles (`r'version\s*=\s*[\'"]([^\'"]+)[\'"]'`), and — more importantly — make the "at least 2 independent sources" check actually require an independent third source (e.g. explicitly fail loudly, rather than silently degrade, if `_read_pyproject_version()` returns `None`), since `pyutilz.__version__`/`version_module.__version__` are not meaningfully distinct sources at all.

### [MEDIUM, needs confirmation] `polars>=0.18` floor pin looks stale against the API surface actually used — pyproject.toml:52, src/pyutilz/data/polarslib.py:20,152-159,642
- **Category**: packaging / dependency-management
- **Problem**: The `polars` extra pins `polars>=0.18` (pyproject.toml:52). `data/polarslib.py:20` unconditionally does `import polars as pl, polars.selectors as cs`, and the file makes heavy use of `.group_by()` (snake_case, lines 152,157,159,696,703) and `.group_by_dynamic()` (line 642). I could not verify the exact polars release that introduced the `polars.selectors` submodule and the `groupby`→`group_by` rename against a live package index (no network access in this sandbox), so I'm flagging this as needs-confirmation rather than a settled fact — but both features are, to my knowledge, notably newer than the 0.18.0 floor, which if correct means `pip install "polars==0.18.0"` (satisfying the declared floor) would raise `ModuleNotFoundError: No module named 'polars.selectors'` or `AttributeError` on `.group_by`/`.group_by_dynamic` before ever reaching user code.
- **Failure scenario** (if the version premise above is confirmed correct): A user pins `polars==0.18.0` per the stated floor, installs `pyutilz[polars]`, and any polarslib function crashes immediately on import or first call.
- **Suggested fix**: Bump the `polars` floor to whatever version actually introduced `polars.selectors` + `group_by`/`group_by_dynamic` (verify against polars' changelog), or add a CI leg that actually installs the pinned floor version and imports polarslib to catch this class of drift going forward.

### [LOW] Stray legacy backup file committed inside the installable package directory — src/pyutilz/__init__.py.old
- **Category**: OSS-practice / repo-hygiene
- **Problem**: `src/pyutilz/__init__.py.old` is tracked in git (confirmed via `git ls-files`) and sits inside `src/pyutilz/`, the exact directory `[tool.setuptools.packages.find]` scans. It contains an old `NotebookColors` enum that predates the current lazy-module-alias design in `__init__.py`. Because it isn't a `.py` file and isn't declared in `[tool.setuptools.package-data]`, it should not actually ship in the built wheel/sdist — this is a repo-cleanliness issue, not a shipped-artifact bug.

- **Suggested fix**: `git rm src/pyutilz/__init__.py.old` (the current `__init__.py` is presumably the canonical version already).

## Things done well

- The extras design is genuinely well thought through where it works: self-referential groups (`dataframes = ["pyutilz[pandas,polars]"]`, `all = ["pyutilz[pandas,polars,database,web,cloud,nlp,llm,system]"]`) avoid version-list duplication/drift, and the `gpu` extra is deliberately excluded from `all` with a clear rationale (CUDA-specific wheel resolution shouldn't be forced on an "everything" install).
- `deptry` triage (pyproject.toml:303-385) is thorough and the per-import rationale is written inline rather than just silenced — including one real bug it caught and fixed along the way (the catboost import-scope fix documented at lines 356-359).
- `tests/test_meta/test_optional_deps_isolation.py`'s core mechanism (a subprocess-isolated `MetaPathFinder` that blocks specific top-level packages) is a genuinely clever, well-engineered way to simulate "extra not installed" without needing separate venvs, and its own docstrings document two real bugs it already caught and fixed (the selenium leak via `pyutilz.system` → `pyutilz.web.browser`, and the nltk leak via `pyutilz.text.tokenizers`) — this report's findings are simply the next increment of exactly that bug class, in places the existing mask doesn't yet reach.
- `CHANGELOG.md:11` already documents the `dataframes` extras group rationale, showing an iterating, self-correcting packaging history rather than a "set once, never revisited" design.
- Version pin comments in the `dev` extra (pyproject.toml:177-191) show real engineering discipline — the `black` marker explains a concrete CI failure it prevents, and the `ruff==0.15.20` exact pin explicitly cites a prior cross-repo drift incident as the reason for pinning exactly rather than with an open range.
- `tests/test_meta/test_version_consistency.py`'s docstring clearly states the exact failure mode it targets and why (useful even though the implementation itself has the gap documented above).

## Investigated, not an issue

- **`match`/`case` statements (Python 3.10+)**: grepped and AST-scanned (`ast.Match` node search) the entire `src/pyutilz` tree — zero hits. No violation of the declared Python 3.8 floor from this syntax.
- **`X | Y` union-type syntax as a real runtime-evaluated annotation**: AST-scanned every `ast.BinOp` with `ast.BitOr` across the tree. Two hits occur in files without `from __future__ import annotations`: `data/polarslib.py:661` (`cs.contains(...) | cs.contains(...)`, a chained polars `Selectors` combination — an object-operator overload, not a type union) and `system/distributed.py:118` (`set(...) - set(...) | set(...)`, a legitimate runtime set-union) — neither is 3.10+-only syntax.
