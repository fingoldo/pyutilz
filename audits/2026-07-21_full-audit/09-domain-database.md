# Domain audit: database (SQL helpers, sqlite, upsert, Delta Lake, Redis)

## Summary

Covered all 7 files in scope: `src/pyutilz/database/__init__.py`, `deltalakes.py`, `redislib.py`, and `db/__init__.py`, `db/sql_helpers.py`, `db/sqlite.py`, `db/upsert.py`. Focus was correctness, efficiency, edge cases (empty results, NULL handling, connection lifecycle, transaction boundaries, upsert conflict resolution, retry-after-disconnect). Every finding below marked "confirmed via live test" was reproduced with a standalone, read-only `python -c` snippet against the actual installed code (no repo files were modified). The historical SQL-injection fix noted in CHANGELOG.md (v0.90, `db.py`) has **not** regressed — `validate_sql_identifier`/`validate_sql_qualified_identifier` are applied consistently at every identifier-splicing site I checked in the current `sql_helpers.py`/`upsert.py`/`db/__init__.py`/`sqlite.py`; no unfixed sibling path was found. However, the module has a cluster of real, previously-unreported correctness bugs unrelated to injection: one documented DML mode (`db_command(mode="update")`) is unconditionally broken, `build_upsert_query`'s history-table feature corrupts caller state via list aliasing, `redislib.rexecute` retries forever including on permanent auth failures, and `insert_sqllite_data` both returns the wrong value on a common call pattern and leaves partial data in an open transaction after a reported failure.

Findings: 5 High, 7 Medium, 7 Low.

## Findings

### [HIGH] `db_command(mode="update", ...)` always builds syntactically-invalid SQL — src/pyutilz/database/db/sql_helpers.py:87-90 (root cause), src/pyutilz/database/db/__init__.py:397-400 (manifests)
- **Category**: correctness
- **Problem**: `construct_templates_and_values()` only builds per-field `key=%s` templates for `mode == "insert"` (unconditional `%s`) or `mode == "select"` (`key + "=%s"`). There is no branch for `mode == "update"`, so for update mode the function still fills `values` correctly but returns an **empty** `templates` list, for both the SET-field call and the WHERE-field call inside `db_command`.
- **Failure scenario**: Any call to `db_command("update", table_name, where_fields=["id"], set_fields=["name"], source={"id": 5, "name": "bob"})` builds `sql = "update  mytable set  where "` (confirmed live — see command below) with `sql_fields_values = ["bob", 5]`. Executing this against Postgres raises a syntax error (`set` immediately followed by `where`, and 2 supplied parameters against 0 `%s` placeholders). Reproduced directly:
  ```
  construct_templates_and_values('update', ['name','age'], {}, {'name':'bob','age':30}, jsonize=True)
  -> values=['bob', 30], templates=[]
  ```
  giving `sql = 'update  mytable set  where '`. `db_command` is a public, documented function (`mode in ["select", "insert", "update"]` is explicitly validated at its entry), so this is a 100%-reproducible break of one of its three supported modes. No internal caller in the repo currently exercises `mode="update"` (`src/pyutilz/system/distributed.py` only uses `"select"`/`"insert"`), so it is currently latent rather than actively crashing shipped code paths — but any external consumer of the `pyutilz` package who uses the documented "update" mode hits this immediately, every time.
- **Suggested fix**: Add a third branch in `construct_templates_and_values` (`elif mode == "update": templates.append(key + "=%s")` — identical shape to `"select"`, since both need `field=%s`), or have `db_command` pass `mode="select"` for both of its `construct_templates_and_values` calls when building an UPDATE's WHERE/SET fragments. Needs a regression test that actually executes/asserts the generated SQL for `mode="update"`, since the current test suite only checks `db_command` is exported (`tests/test_db_extra.py:289`), not its generated SQL shape.

### [HIGH] `build_upsert_query`'s `history_fields` gets silently mutated by list aliasing, corrupting the generated history-table INSERT — src/pyutilz/database/db/upsert.py:178-185
- **Category**: correctness
- **Problem**: `returning_fields = history_fields` binds `returning_fields` to the *same* list object as the local `history_fields` (which, unless the caller happened to pass `None`, is literally the caller's own list — there is no `.copy()`). The subsequent `returning_fields += [hash_field]` is `list.__iadd__`, an in-place mutation, so it also mutates `history_fields` (and the caller's original list, if they passed one in). `history_fields` is used again further down (line ~194, `history_fields_final = [... for field in history_fields]`, and line ~196, `select {','.join(['u.'+field for field in history_fields])}`) to build the actual `INSERT INTO {history_table_name}(...)` statement — so the hash field(s), which are only supposed to appear in the `fresh_data` CTE's `RETURNING` list (needed for the old/new hash diff), leak into the history-table's column list too.
- **Failure scenario**: Confirmed live —
  ```python
  build_upsert_query(fields_names=['id','name','md5hash'], table_name='mytable',
                      conflict_fields=['id'], history_table_name='mytable_history',
                      history_fields=['id','name'], hash_fields='md5hash')
  ```
  produces `...,changed_data as (insert into mytable_history(id,name,md5hash) select u.id,u.name,u.md5hash from fresh_data u ...` — `md5hash` was never requested as a history column (`history_fields=["id","name"]`), but it is now inserted into `mytable_history`. If that table's schema only has `id,name` columns (as the caller's `history_fields` argument implies), executing this raises `column "md5hash" of relation "mytable_history" does not exist`; if the table coincidentally has a same-named column for something else, wrong data is silently written into it. Additionally, since no defensive copy is made, if the caller reuses the same `history_fields` list object across multiple `build_upsert_query()` calls (a very natural pattern — e.g. a module-level constant list), the corruption is permanent on that object after the first call, silently affecting every later use of that list anywhere else in the caller's code, not just this function.
- **Suggested fix**: `returning_fields = list(history_fields)` (or `history_fields.copy()`) before the `+=`, so the RETURNING-list extension for the fresh-data CTE never leaks back into the caller's `history_fields`.

### [HIGH] `redislib.rexecute` retries forever on `ConnectionError`, including permanent authentication failures — src/pyutilz/database/redislib.py:62-76
- **Category**: correctness / retry-after-disconnect
- **Problem**: The `while True:` retry loop has no attempt cap (unlike `basic_db_execute`'s explicit `max_retries` circuit breaker in `db/__init__.py`). Worse, `redis.exceptions.AuthenticationError` is a **subclass** of `redis.exceptions.ConnectionError` in redis-py (confirmed against the installed `redis==8.0.1`: `issubclass(AuthenticationError, ConnectionError) == True`), and `RedisConnectionError` here is exactly that `ConnectionError`. A permanent, non-transient failure (wrong password) is therefore caught by the same "retry with jittered backoff" branch as a genuinely transient network blip.
- **Failure scenario**: `rconnect(host, port, db, wrong_password)` followed by any `rexecute("get", "key")` loops forever: `method(*args, **kwargs)` raises `AuthenticationError` (a `RedisConnectionError`), the loop logs it and sleeps `1*random()` seconds, then retries the *same* doomed call indefinitely — no exception ever propagates to the caller, no way to detect the outage other than watching logs grow unbounded, and the calling thread never returns. Contrast with `basic_db_execute`'s Postgres path, whose comment explicitly states the goal is "to prevent infinite retry loops" via a bounded `max_retries` — this module has no equivalent.
- **Suggested fix**: Add a `max_retries` parameter (mirroring `basic_db_execute`) that raises after N attempts, and/or explicitly exclude `AuthenticationError` (and other clearly-permanent `ConnectionError` subclasses) from the retry branch so they propagate immediately.

### [HIGH] `insert_sqllite_data` leaves partially-inserted rows in an open, uncommitted transaction on failure — src/pyutilz/database/db/sqlite.py:68-78
- **Category**: correctness / transaction boundaries
- **Problem**: The `except Exception` handler logs the error and returns `0`, but never calls `conn.rollback()`. `cursor.executemany()` on `sqlite3` executes each parameter set sequentially; if row *k* violates a constraint, rows `1..k-1` have already been executed against the connection and remain pending in an open transaction, neither committed nor rolled back.
- **Failure scenario**: Confirmed live — inserting `[{'id':1,'name':'a'}, {'id':2,'name':'b'}, {'id':3,'name':None}, {'id':4,'name':'d'}]` into a table with `name text not null` raises on row 3; `insert_sqllite_data` returns `0` (implying "nothing happened"), but `conn.in_transaction` is `True` and `select * from t` on that same connection already shows `[(1,'a'), (2,'b')]` pending. Only an explicit, external `conn.rollback()` clears them. If any later, unrelated code path on the same connection calls `conn.commit()` for a different reason (a very plausible pattern for a shared `sqlite3.Connection` reused across multiple `insert_sqllite_data` calls, which itself calls `conn.commit()` on success), those two rows the caller was told never got inserted are silently committed for real.
- **Suggested fix**: `conn.rollback()` in the `except` branch before returning, so a reported failure actually leaves the DB (and the transaction) in the state the return value implies.

### [HIGH] `insert_sqllite_data` returns `None` instead of the row count on a successful insert when `verbose=0` — src/pyutilz/database/db/sqlite.py:68-74
- **Category**: correctness
- **Problem**: `return n` is nested *inside* `if verbose:`, not after it. When `verbose` is falsy (the function's own recommended way to silence per-call logging), the successful path falls off the end of the `try` block with no `return` statement, so the function implicitly returns `None` — even though the commit succeeded and `n` rows really were inserted.
- **Failure scenario**: Confirmed live —
  ```python
  insert_sqllite_data('t', [{'id':1,'name':'a'},{'id':2,'name':'b'}], ['id','name'], cur, conn, verbose=0)
  -> None      # rows WERE committed successfully
  insert_sqllite_data('t', [{'id':3,'name':'c'}], ['id','name'], cur, conn, verbose=1)
  -> 1         # same code path, verbose=1, correct count
  ```
  A caller accumulating totals (`total += insert_sqllite_data(..., verbose=0)`) crashes with `TypeError: unsupported operand type(s) for +=: 'int' and 'NoneType'` on a fully successful insert; a caller checking `if not insert_sqllite_data(...): logger.warning("nothing inserted")` gets a false-positive warning after real data was committed.
- **Suggested fix**: Move `return n` outside the `if verbose:` block (keep the logging conditional, not the return).

### [MEDIUM] `connect_to_db(m_db_flavor="mysql")` never sets `conn`/`cur`, silently breaking the entire cursor-based API for MySQL — src/pyutilz/database/db/__init__.py:188-193
- **Category**: correctness / architecture
- **Problem**: The function's top-level assert (`assert db_flavor in ("postgres", "mysql")`, line 149) documents MySQL as a first-class supported flavor. But the `elif db_flavor == "mysql":` branch only builds `conn_alchemy` via SQLAlchemy — it never assigns the module-level `conn`/`cur` globals that the entire cursor-based API (`get_cursor`, `basic_db_execute`, `safe_execute`, `get_table_fields`, `db_command`, `log_to_db`, `EnsurePgTableExists`, `ReadTableIntoDic*`, `GetIdByKeyFieldAndInsertIfNeeded`, `create/delete_postgres_range_partitions`, etc.) depends on.
- **Failure scenario**: `connect_to_db(..., m_db_flavor="mysql")` followed by any call that eventually reaches `get_cursor()` (e.g. `safe_execute("select 1")`) hits `assert conn is not None, "get_cursor() requires connect_to_db() to have been called first"` and raises `AssertionError` — even though `connect_to_db()` *was* called, with the exact flavor the assert list explicitly claims to support. Only the SQLAlchemy-only helpers (`select()`, `execute_alchemy()`, `explain_table()` for mysql) actually work.
- **Suggested fix**: Either build a real `pymysql` `conn`/`cur` pair in the mysql branch (mirroring the postgres branch), or narrow the flavor assert / module docstring to state that only `execute_alchemy`/`select`/`explain_table` support MySQL, so the gap is documented rather than silently discovered via an AssertionError.

### [MEDIUM] `build_upsert_query` silently ignores `conflict_fields` (falls back to a bare, unscoped `on conflict do nothing`) whenever `on_conflict_update_fields` is empty — src/pyutilz/database/db/upsert.py:153-176
- **Category**: correctness / upsert conflict-resolution
- **Problem**: The branch that emits `on conflict ({conflict_fields}) do update set ...` is only reached when `custom_onconflict` OR (`conflict_fields` non-empty **and** `on_conflict_update_fields` non-empty). If the caller wants "skip duplicates on this specific key, but still error on other unique-constraint violations" — i.e. `conflict_fields=["id"]`, `on_conflict_update_fields=[]` (the natural way to express DO NOTHING targeted at one constraint) — the `conflict_fields` argument is completely ignored and the code falls to the `else` branch, `fresh_query += "on conflict do nothing "` (no column/constraint list at all).
- **Failure scenario**: Confirmed live — `build_upsert_query(fields_names=['id','name','md5hash'], table_name='mytable', conflict_fields=['id'], history_table_name='mytable_history', history_fields=['id','name'], hash_fields='md5hash')` (no `on_conflict_update_fields` passed) generates `insert into mytable (...) select ... from data on conflict do nothing ...` — the bare form. A bare `ON CONFLICT DO NOTHING` in Postgres matches **any** unique/exclusion constraint violation on the table, not just the one implied by `conflict_fields`. If `mytable` also has a unique constraint on, say, `email`, a genuine duplicate-email bug that the caller would want surfaced as an error is now silently swallowed as "conflict, do nothing" instead.
- **Suggested fix**: When `on_conflict_update_fields` is empty but `conflict_fields` is non-empty, emit `on conflict ({','.join(conflict_fields)}) do nothing` instead of the bare form, so only the intended constraint is targeted.

### [MEDIUM] `EnsurePgTableExists`'s UUID-default check is case-sensitive after a case-insensitive validation, silently dropping the default generator — src/pyutilz/database/db/__init__.py:531-538
- **Category**: correctness / edge-case
- **Problem**: `if sAutocreateIdTypeName.lower() not in ("smallserial serial bigserial uuid".split()): raise ValueError(...)` validates case-insensitively, but the very next line, `default_gen = " default gen_random_uuid()" if sAutocreateIdTypeName == "uuid" else ""`, compares the **original, non-lowered** value.
- **Failure scenario**: `EnsurePgTableExists("mytable", sAutocreateIdTypeName="UUID")` passes validation (since `"UUID".lower() == "uuid"` is in the whitelist) but `"UUID" == "uuid"` is `False`, so `default_gen` stays `""`. The generated DDL is `create table mytable (id UUID primary key ,name text, ...)` — no `default gen_random_uuid()`. Any subsequent insert that relies on the DB auto-generating the id (omitting it from the INSERT column list, which is exactly the point of `sAutocreateIdTypeName`) fails with a NOT NULL constraint violation instead of silently getting a generated UUID.
- **Suggested fix**: Compare the lowered value: `default_gen = " default gen_random_uuid()" if sAutocreateIdTypeName.lower() == "uuid" else ""`.

### [MEDIUM] `db_command` crashes or builds malformed SQL on empty/missing `where_fields` — src/pyutilz/database/db/__init__.py:361-380, sql_helpers.py:59-91
- **Category**: correctness / edge-case
- **Problem**: The up-front sanity check only rejects the case where *both* `where_fields` and `set_fields` are `None` (line 361-363). For `mode="select"` (or `"update"`), only `where_fields` is actually consulted to build the WHERE clause — but a caller can pass `where_fields=None, set_fields=[...]` and pass the check, then crash; or pass `where_fields=[]` (empty list, not `None`) and pass the check silently.
- **Failure scenario**: Confirmed live — `construct_templates_and_values('select', None, {}, {}, jsonize=True)` raises `TypeError: 'NoneType' object is not iterable` (reachable from `db_command(mode="select", set_fields=["x"], where_fields=None, table_name="t")`, since the sanity check only requires *one* of the two to be non-None). Separately, `db_command(mode="select", where_fields=[], table_name="t")` passes the check (`[] is not None`) and builds `sql = "select * from  t where " + " and ".join([])` → `"select * from  t where "` — a trailing `where` with no condition, which Postgres rejects as a syntax error at execution time rather than failing fast with a clear validation message.
- **Suggested fix**: For `select`/`update` modes, explicitly require `where_fields` to be a non-empty list (reject `None` and `[]` both) before calling `construct_templates_and_values`, with a clear `ValueError`/log message instead of deferring to a downstream `TypeError` or a Postgres syntax error.

### [MEDIUM] `basic_db_execute`'s "cursor already exists" recovery branch bypasses the retry circuit breaker and can close the wrong cursor — src/pyutilz/database/db/__init__.py:254-307
- **Category**: correctness / retry-after-disconnect
- **Problem**: The loop's circuit breaker (`retry_count < max_retries`) is only incremented in the `except (OperationalError, InterfaceError)` branch. The generic `except Exception as e:` branch that special-cases "cursor ... already exists" errors (line 300-307) never touches `retry_count`, so if that specific error condition keeps recurring (e.g. a named/server-side cursor whose name collides deterministically because a prior transaction left it open), the `while` loop can iterate indefinitely without ever hitting the 5-retry cap the comment at line 254 explicitly says exists "to prevent infinite retry loops." Additionally, since the failing `cur = get_cursor(...)` assignment never completes when `get_cursor()` itself raises, `cur` still holds whatever it was set to on the *previous* iteration/call; `if cur is not None: cur.close()` in this branch therefore closes that stale, unrelated cursor object rather than the one that actually collided.
- **Failure scenario**: A named cursor (`cursor_name="mycursor"`) already open server-side (e.g. left over from a prior failed transaction that was never cleaned up) causes `conn.cursor(..., name="mycursor")` to raise `psycopg2.errors.DuplicateCursor` ("cursor "mycursor" already exists"), which is a `ProgrammingError` and therefore not caught by the `OperationalError`/`InterfaceError`/`DuplicateTable`/`InternalError` branches — it falls into the generic handler. Since named cursors are never cached (`if "_named" not in cursor_type: cursors[cursor_type] = cur`), `cursors.pop(cursor_type, None)` is a no-op, the collision is never actually resolved, and the loop retries the identical failing call forever (no `retry_count` increment), each time closing whatever the previous, unrelated `cur` value was.
- **Suggested fix**: Increment `retry_count` (and apply the same circuit breaker) in this branch too; only close `cur` if it is actually the cursor implicated in this specific exception (e.g. capture the return of `get_cursor()` in a fresh local variable before doing anything else with it, or track cursor identity explicitly).

### [MEDIUM] `execute_alchemy` silently swallows the failure after exhausting all retries — src/pyutilz/database/db/__init__.py:753-772
- **Category**: correctness / retry-after-disconnect
- **Problem**: The `while n < max_retries:` loop retries any exception (including permanent ones like a SQL syntax error) up to `max_retries` (default 3) times, logging via `logger.exception` and sleeping 3s each time — but once the loop's condition becomes false (all attempts exhausted), execution simply falls through to the end of the function with no `return`/`raise`, so the function returns `None` with no error surfaced to the caller at all.
- **Failure scenario**: `execute_alchemy("UPDTAE users SET x=1")` (a typo, or any permanently-broken/permission-denied statement) fails identically on all 3 attempts — each failure is transient-shaped code treating a non-transient error the same way — then the function returns `None` silently. A caller with no logging visibility (or logs rotated away) has no indication the statement never executed. Contrast with `basic_db_execute`, which explicitly does `if retry_count >= max_retries: logger.error(...); raise` on the equivalent case.
- **Suggested fix**: Re-raise the last exception (or a wrapping exception) after the loop exits without a successful `break`, matching `basic_db_execute`'s pattern.

### [MEDIUM] `safe_delta_write`'s lock file is keyed only on `basename(path)`, so unrelated tables with the same name serialize against each other — src/pyutilz/database/deltalakes.py:75-81
- **Category**: efficiency / architecture
- **Problem**: `lock_file = os.path.join(tempfile.gettempdir(), f"{os.path.basename(path).replace('/', '_')}{lock_suffix}")` discards the directory component entirely. Any two distinct Delta tables that happen to share a basename (a very plausible layout, e.g. `/data/region-us/orders` and `/data/region-eu/orders`, or `/data/2025/orders` vs `/data/2026/orders`) map to the exact same OS-wide lock file in the shared temp directory.
- **Failure scenario**: A process writing to `/data/region-us/orders` and a concurrent, completely unrelated process writing to `/data/region-eu/orders` contend for the same `FileLock`, forcing one to block for up to `lock_timeout` (default 1200 seconds = 20 minutes) even though the two tables have nothing to do with each other and there is no real data-race between them.
- **Suggested fix**: Derive the lock filename from a hash (or the full normalized path, sanitized) of `path` rather than just its basename, e.g. `hashlib.sha256(os.path.abspath(path).encode()).hexdigest()`.

### [LOW] `safe_delta_write` returns `None` indistinguishably on both "lock timed out, operation skipped" and "operation ran and legitimately returned None" — src/pyutilz/database/deltalakes.py:78-86
- **Category**: edge-case / API design
- **Problem**: On `Timeout`, the function logs a warning and falls off the end of the `try/except` returning `None` — with no exception, no sentinel, and no way for the caller to tell this apart from a successful `delta_op_func()` call that itself returned `None` (which `write_deltalake()` does, per the module's own docstring example).
- **Failure scenario**: A caller doing `result = safe_delta_write(path, write_market_ads); if result is None: ...` cannot distinguish "the write succeeded" (write_deltalake normally returns None) from "the write was skipped entirely because another process held the lock for 20 minutes" — a silently-dropped write looks identical to a normal successful append.
- **Suggested fix**: Raise a dedicated exception (or return a distinguishable sentinel/result object) on lock timeout instead of swallowing it into a bare `None` return.

### [LOW] `insert_sqllite_data`'s reserved-keyword quoting only covers the exact string `"GROUP"` — src/pyutilz/database/db/sqlite.py:58
- **Category**: edge-case
- **Problem**: `columns_str = ", ".join([f'"{col}"' if col == "GROUP" else col for col in columns])` is a case-sensitive, single-keyword special case. SQLite's reserved-keyword matching is case-insensitive, and there are many other reserved words besides `GROUP` (`ORDER`, `TABLE`, `SELECT`, `WHERE`, `INDEX`, etc.).
- **Failure scenario**: Confirmed live — a column literally named `"group"` (lowercase, the natural spelling) is **not** caught by `col == "GROUP"` and still produces `INSERT INTO t (id, group) VALUES (?, ?)`, which sqlite3 rejects with `OperationalError: near "group": syntax error`; the function's own `except` handler then reports this as "Could not insert data" and returns `0` (compounding with the two `insert_sqllite_data` findings above). The same failure reproduces for `"order"` as a column name, which isn't special-cased at all in any casing.
- **Suggested fix**: Always double-quote every column identifier (`f'"{col}"'` unconditionally) rather than special-casing one keyword — quoting is always valid in SQLite regardless of whether the name happens to be reserved.

### [LOW] Zero-padding format spec is a no-op in partition table naming — src/pyutilz/database/db/__init__.py:687, 711
- **Category**: correctness / cosmetic
- **Problem**: `f"z_{table_name}_y{d.year:0000d}m..."` uses format spec `"0000d"`. Confirmed live: this does **not** zero-pad — `f"{5:0000d}"` produces `"5"`, not `"0005"` (the leading `0` is consumed as the zero-pad *flag*, and the remaining `"000"` is parsed as *width* = 0, which imposes no minimum width). `f"{5:04d}"` (the presumably-intended spec) correctly produces `"0005"`.
- **Failure scenario**: For any `from_date`/`to_date` with `d.year < 1000` (e.g. synthetic/test fixtures using small years), `create_postgres_range_partitions`/`delete_postgres_range_partitions` generate a partition table name segment with fewer than 4 digits (`z_t_y5m03w1d15` instead of `z_t_y0005m03w01d15`), which would not lexically sort/align the way a consistently zero-padded scheme intends, and a `create` followed by a `delete` call using the same (buggy but consistent) format would still round-trip correctly against each other — real-world calendar years (2024+) are unaffected since they're already 4 digits.
- **Suggested fix**: Use `{d.year:04d}` instead of `{d.year:0000d}`.

### [LOW] `connect_to_db`'s startup retry loop has no distinction between transient and permanent failures — src/pyutilz/database/db/__init__.py:151-206
- **Category**: robustness (needs confirmation this is unintentional — an unbounded "wait for DB at startup" loop can be a deliberate design choice)
- **Problem**: `while True: try: ... except Exception as e: logger.exception(e); ...; sleep(5)` retries forever on *any* exception, including permanent misconfiguration (wrong password, wrong host, wrong database name) — there is no cap and no way to distinguish "DB is temporarily down, keep waiting" from "these credentials will never work."
- **Failure scenario**: If credentials are rotated/revoked while the process is calling `connect_to_db` (either at startup or via `basic_db_execute`'s reconnect-on-error path), the process logs an exception and sleeps 5 seconds, forever, with no upper bound and no way for an orchestrator/health-check to detect a genuinely unrecoverable state versus normal startup-ordering delay.
- **Suggested fix**: Optionally accept a `max_attempts`/`max_wait` parameter, or at least classify a subset of exceptions (e.g. `psycopg2.OperationalError` with an "authentication failed" message) as non-retryable and re-raise them.

### [LOW] `u()`/`nu()` assume a string input with no type check — src/pyutilz/database/db/sql_helpers.py:94-111
- **Category**: edge-case / robustness
- **Problem**: `u(str_val, symb="'")` does `symb + str_val.replace(symb, symb * 2) + symb`; `nu()` does `len(str_val) == 0`. Neither validates that `str_val` is actually a `str` (or `None`). Passing a non-string, non-None value (e.g. an `int` or `date`) raises `AttributeError: 'int' object has no attribute 'replace'` (for `u()`) or `TypeError: object of type 'int' has no len()` (for `nu()`) instead of a clear, purpose-specific error.
- **Failure scenario**: A caller building a literal for a numeric or date value with `u(some_int)` (plausible, since the function name/docstring don't make the str-only constraint obvious to a new caller) gets an opaque `AttributeError` instead of either working correctly (e.g. via `str(str_val)` coercion) or a clear `TypeError("u() requires a str or None")`.
- **Suggested fix**: Either coerce non-None inputs with `str()` before quoting, or raise an explicit, descriptive `TypeError` for non-str/non-None input.

### [LOW] Dead code: `rev_on_conflict_update_values` is computed but never used — src/pyutilz/database/db/upsert.py:111-117
- **Category**: efficiency / maintainability
- **Problem**: 
  ```python
  rev_on_conflict_update_values = {}
  if len(on_conflict_update_values) > 0:
      for _field in on_conflict_update_values.keys():
          # assert _field in fields_names
          pass
      for field, value in on_conflict_update_values.items():
          rev_on_conflict_update_values[field] = value.replace("excluded.", "")
  ```
  `rev_on_conflict_update_values` is never referenced again anywhere in the function. The first loop is a literal no-op (`pass`) with a commented-out assert. This reads like an abandoned refactor.

### [LOW] `check_if_pg_table_exists` splices manually-escaped literals into an f-string instead of using `%s` parameter binding — src/pyutilz/database/db/__init__.py:513-522
- **Category**: correctness / consistency (adjacent to the dedicated SQL-injection audit, noted here for correctness-style consistency)
- **Problem**: `safe_execute(f"""SELECT EXISTS (...WHERE table_schema = {u(schema_name)} AND table_name={u(table_name)})""")` manually quotes/escapes via `u()` and splices the result into the query text, rather than using `safe_execute(sql, (schema_name, table_name))` with `%s` placeholders — which the very next function in the same file, `log_to_db`, does correctly for its parameters. `u()`'s escaping (doubling embedded `'`) is narrower than full driver-level parameter binding (e.g. it does not account for encoding/backslash edge cases the driver handles for you).
- **Failure scenario**: Purely a style/consistency gap under normal inputs; flagged for the SQL-injection audit to weigh, but from a correctness standpoint it's inconsistent with the rest of the module's convention of preferring `%s` binding wherever the value is a comparison target rather than an identifier.

## Things done well

- **Native atomic upsert via `INSERT ... ON CONFLICT`** (`upsert.py`): `build_upsert_query` builds a single-statement CTE-based upsert (`with data(...) as (VALUES %s), fresh_data as (insert ... on conflict ... )`) rather than a check-then-insert pattern — this is the *correct* way to be safe under concurrent writers (Postgres resolves the conflict atomically at the row level), and is a real strength once the aliasing bug above is fixed.
- **`InternalError` is deliberately re-raised, with an explanatory comment**, rather than silently swallowed or auto-rolled-back (`db/__init__.py:292-299`) — the comment explains exactly why ("do not silently fall through to an implicit None... or spin forever"), which is good defensive-code hygiene.
- **`_SQL_IDENTIFIER_RE` is pre-compiled at module scope with a measured rationale** (`sql_helpers.py:27-30`, ~2.3x faster than `re.match` per the inline timeit comment) — identifier validation is on a genuinely hot per-query path, and the optimization is documented rather than just applied silently.
- **`validate_sql_identifier`/`validate_sql_qualified_identifier` are applied consistently** at essentially every table/column-name splicing site across `db/__init__.py`, `sqlite.py`, and `upsert.py` that I checked — no unfixed sibling of the CHANGELOG's historical (v0.90) SQL-injection issue was found; the raw-SQL escape hatches that remain (`select()`, `execute_alchemy()`, `showcase_table()`'s `condition` param, `ReadTableIntoDicReversed`'s `sCondition`) are explicitly documented in their docstrings as "trusted input only," which is the right way to carve out an intentional exception.

- **`psycopg2.sql.Identifier`/`sql.SQL` used correctly** in `read_unique_table_field` and `create_enum_from_table` — proper driver-level identifier quoting rather than manual validation, a more robust pattern than the regex-based `validate_sql_identifier` used elsewhere (though both approaches coexist deliberately, presumably for historical/incremental-adoption reasons).
- **`safe_delta_write` clearly separates local-vs-remote-store handling** and explicitly documents (via a warning log) that remote-path writes proceed without a lock, rather than pretending to offer a guarantee it can't provide over S3/Azure.

## Investigated, not an issue

- **`validate_sql_qualified_identifier` looked unused within `db/__init__.py`/`upsert.py`** (only imported/re-exported, never called in those files) — turned out to be a false alarm: it's actively used in `src/pyutilz/dev/logginglib.py:210` and has its own dedicated tests in `tests/test_db_extra.py:74-87`. - **`check_if_pg_table_exists`'s `if res: return res[0][0]`** looked suspicious at first glance (worried a falsy `False` payload might be swallowed by the truthiness check) — confirmed this is actually correct: `res` is the *list* `[(False,)]`, which is truthy as a non-empty list regardless of the boolean value inside it, so `False` is returned correctly, not swallowed.
