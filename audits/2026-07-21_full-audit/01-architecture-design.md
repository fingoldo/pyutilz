# Architecture & Design Audit — pyutilz

## Summary

I read every `__init__.py` under `src/pyutilz` (top-level and all 11 subpackages), the three registry
implementations (`llm/factory.py`, `dev/code_audit/registry.py`, `performance/kernel_tuning/registry.py`),
the exception modules, the backward-compat alias map and its meta-test, and traced every
`from pyutilz.X import Y` cross-subpackage edge in `src/pyutilz` to check import direction against the
"install only what you need" promise in `pyproject.toml`/`README.md`. Every dependency-chain claim below
was verified empirically with a subprocess-style `sys.meta_path` import-blocking probe (the same technique
`tests/test_meta/test_optional_deps_isolation.py` uses), not just by reading the import list, so the
"masked deps → import fails" outcomes are reproduced facts, not inferred.

The headline finding is that the "install only what you need" promise is broken through a chain of
**undocumented, unconditional cross-domain imports that the existing optional-dep-isolation meta-test
does not catch because it only probes package `__init__.py` files, not the leaf submodules real callers
actually import**. Most severe: `pyutilz.core.pythonlib` — the single most widely-imported "core" module,
and the one README.md explicitly promises has "zero hard dependencies" — cannot itself be imported without
`numba`, `joblib`, and `portalocker` (all `[system]`-extras-only). Several further instances of the same
bug class exist (`text.strings` needs pandas/numpy undocumented; `text.tokenizers` and `dev.logginglib`
unconditionally need the `[database]` stack; `data.polarslib`, declared under `[polars]` only, needs the
entire `[system]` stack). I also compared the three registry patterns, checked exception-hierarchy
consistency across domains, verified the "24-entry" alias-map claim in TESTING.md, and compared
pandaslib/polarslib naming for the same conceptual operations.

Findings: 1 Critical, 4 High, 4 Medium, 2 Low. Listed most severe first.

## Findings

### [Critical] `pyutilz.core.pythonlib` — the library's most depended-upon "core" module — cannot import without `numba`/`joblib`/`portalocker`, contradicting README's explicit "zero hard dependencies" promise — src/pyutilz/core/pythonlib.py:31,37,869
- **Category**: architecture / packaging boundary
- **Problem**: `README.md:14` states "The core has zero hard dependencies" and `README.md:21` documents `pip install pyutilz # core only, no hard deps`. `src/pyutilz/core/pythonlib.py` is exactly that "core" module — dozens of other files across every domain (`filemaker.py`, `data/pandaslib/_common.py`, `database/db/__init__.py`, `dev/logginglib.py`, `text/strings/__init__.py`, `web/browser.py`, `system/hardware_monitor.py`, `system/gpu_dispatch.py`, `performance/kernel_tuning/cache/__init__.py`, `system/distributed.py`) do `from pyutilz.core.pythonlib import X`. Yet the module itself does `from numba import njit` (line 31, decorates one function at line 335), uses `joblib.dump`/`joblib.load` as literal default-argument values for two classes (lines 795, 820 — evaluated at import time, not deferrable by the usual "import inside the function" trick), and `import portalocker` (line 869, used inside `open_safe_shelve`). All three packages are declared ONLY under the `[system]` extras group in `pyproject.toml` (`numba>=0.56`, `joblib>=1.2`, `portalocker>=2.0`), not under any "core" dependency set (there is no "core" extras group — core is supposed to need nothing).
- **Failure scenario**: A user follows the README verbatim: `pip install pyutilz` (bare). They then try any pyutilz functionality that isn't `safe_pickle` — e.g. `from pyutilz.core.pythonlib import ensure_installed` (the very helper meant to guide users through missing installs) or `import pyutilz.text.strings` (transitively imports `pyutilz.core.pythonlib`) — and get `ModuleNotFoundError: No module named 'numba'` at import time. Verified directly: masking `numba`/`joblib`/`portalocker` (plus the rest of `[system]`) and running `import pyutilz.core.pythonlib` raises `ImportError: (masked) numba` immediately.
- **Suggested fix**: Move the `njit`-decorated helper and the two joblib-default-arg classes into a submodule that's imported lazily (or guard with `try/except ImportError` + a runtime fallback, matching the pattern already used elsewhere in this same file for `catboost`/`IPython`/`nvidia`, per the deptry per-rule-ignore comments at `pyproject.toml:355-363`). `portalocker.Lock` is trivially deferrable (function-local import in `open_safe_shelve`). Extend `tests/test_optional_deps_isolation.py`'s `_ALWAYS_IMPORTABLE_SUBPACKAGES` to include the leaf module `pyutilz.core.pythonlib` itself (not just the `pyutilz.core` package, whose `__init__.py` does no eager imports and therefore never exercises this).

### [High] `pyutilz.text.strings` unconditionally imports pandas + numpy — undocumented, breaks `[nlp]`-only installs for `core.image`, `text.tokenizers`, `text.similarity` — src/pyutilz/text/strings/__init__.py:21
- **Category**: architecture / packaging boundary
- **Problem**: `src/pyutilz/text/strings/__init__.py:21` does `import pandas as pd, numpy as np` unconditionally at module scope. `pyproject.toml`'s `nlp` extras group (lines 109-122) documents exactly three other unconditional imports in this same domain (`text/strings/webtext.py`'s beautifulsoup4/inflect/emoji-data-python) but says nothing about pandas/numpy here — an inconsistency in the very comment style the project otherwise uses to flag "this unconditional import is a real, declared requirement." `pandas`/`numpy` are declared only under the `pandas` extras group. Three other modules import from `text.strings` and are therefore silently pulled into this same requirement: `src/pyutilz/core/image.py:32`, `src/pyutilz/text/tokenizers.py:28-38`, `src/pyutilz/text/similarity.py:17`.
- **Failure scenario**: `pip install pyutilz[nlp]` (the documented install for the tokenizers/NLP domain per `README.md:26`) does not include pandas/numpy. `import pyutilz.text.strings` (or `pyutilz.text.tokenizers`, or `pyutilz.core.image`) then raises `ModuleNotFoundError: No module named 'numpy'`. Verified directly: masking `pandas`/`numpy`/`pyarrow` and running `import pyutilz.text.strings` fails with `ImportError: (masked) numpy`.
- **Suggested fix**: Either declare pandas/numpy as real requirements of the `nlp` extras group (with the same "why" comment style used elsewhere in `pyproject.toml`), or — better — determine which of the re-exported symbols in `text/strings/__init__.py` actually need pandas/numpy and confine those imports to the specific submodule that needs them rather than the package `__init__.py` that every consumer transitively pays for.

### [High] `pyutilz.system/__init__.py`'s eager multi-submodule import forces the full `[system]` extras stack on `core.serialization` and `data.polarslib`, neither of which declares it — src/pyutilz/system/__init__.py:18, src/pyutilz/core/serialization.py:27, src/pyutilz/data/polarslib.py:28
- **Category**: architecture / packaging boundary
- **Problem**: `src/pyutilz/system/__init__.py:18` does `from . import parallel, monitoring, hardware_monitor, system` eagerly — this is the *exact* bug class the module's own comment (lines 8-17) documents having partially fixed for `distributed`/`[web]` on 2026-07-09 ("Importing it eagerly here forced every caller of an unrelated pyutilz.system submodule ... to have [web] installed too"), but the fix only special-cased `distributed`; `parallel`, `monitoring`, and `hardware_monitor` remain eagerly bundled together. Two concrete undocumented victims: `src/pyutilz/core/serialization.py:27` does `from pyutilz.system import system` purely to call `system.ensure_dir_exists()` (one filesystem helper), and `src/pyutilz/data/polarslib.py:28` does `from pyutilz.system.system import clean_ram` (used at 8 call sites) — both trigger the full package init and thus need `psutil`, `numba`, `joblib`, `tqdm`, `GPUtil`, `py-cpuinfo`, `xmltodict`, `Pillow`, `scipy` (the entire `[system]` group). Neither module is declared to need `[system]` anywhere in `pyproject.toml`; `polarslib.py` is declared under `[polars]` only (`polars>=0.18`), which `README.md:22` documents as `pip install pyutilz[dataframes] # pandas + numpy + pyarrow + polars`.
- **Failure scenario**: A user follows `README.md:22` verbatim, `pip install pyutilz[dataframes]`, then does `import pyutilz.data.polarslib` (or via the `pyutilz.polarslib` back-compat alias) — `ModuleNotFoundError: No module named 'psutil'`. Verified directly for both `pyutilz.core.serialization` and `pyutilz.data.polarslib` by masking the `[system]` group's top-level packages.
- **Suggested fix**: Split `pyutilz/system/system/` further so `ensure_dir_exists`/`clean_ram`-class filesystem/memory helpers that don't need heavy deps aren't bundled with `hardware_monitor`/`parallel` in one eager `from . import`, OR make `parallel`/`monitoring`/`hardware_monitor` lazy via the same PEP 562 `__getattr__` pattern already used for `distributed` two lines below. At minimum, document the real requirement inline in `pyproject.toml` (the project's own established convention) for whichever extras group `core.serialization` and `data.polarslib` actually need.

### [High] `pyutilz.text.tokenizers` unconditionally imports `pyutilz.database.db`, directly contradicting the file's own adjacent comment that DB access is "deferred to call site" — src/pyutilz/text/tokenizers.py:22-25
- **Category**: architecture / packaging boundary
- **Problem**: Lines 22-24 read: *"psycopg2 + pyutilz.database are deferred to call site - tokenizers.py is loaded by mlframe via the pyutilz.text re-export chain; not every consumer needs the database stack just to use string utilities below."* Line 25 immediately below does `from pyutilz.database import db` — a plain, unguarded, module-top-level import, i.e. the opposite of deferred. `pyutilz.database.db.__init__` (`src/pyutilz/database/db/__init__.py:32,35,37`) unconditionally imports `sqlalchemy`, `dateutil.relativedelta`, and `psycopg2` — none declared for the `nlp` extras group. The module's own author clearly understood the principle and even applied it correctly nine lines later for `nltk` (lines 56-60, guarded `try/except`) and again at line 200 (`import psycopg2.extras` deferred inside `tokenize_db_reviews()`, with a comment explaining exactly why) — but the top-level `db` import at line 25 defeats that later deferral anyway, since `pyutilz.database.db`'s own `__init__.py` imports `psycopg2` unconditionally regardless.
- **Failure scenario**: `pip install pyutilz[nlp]` (documented at `README.md:26`), then `from pyutilz.text.tokenizers import AdvancedTokenizer` (the exact usage the file's own comment says should work "to use string utilities" without the database stack) → `ModuleNotFoundError: No module named 'sqlalchemy'`. Verified directly: masking `sqlalchemy`/`psycopg2`/`pymysql` and importing `pyutilz.text.tokenizers` fails immediately with `ImportError: (masked) sqlalchemy`. Not caught by CI: `tests/test_meta/test_optional_deps_isolation.py::test_tokenizers_module_imports_without_nlp_group_deps` (line 134-156) masks only the `nlp` group's deps when testing this exact import — it never masks the `database` group against this target, so the gap is invisible to the existing gate.
- **Suggested fix**: Move `from pyutilz.database import db` into the one function that uses it (`tokenize_db_reviews`, which already lazily imports `psycopg2.extras` right next to where `db.safe_execute` is called at line 203) — mirroring the `spacy`/`nltk` guard pattern already present in the same file. Add a `test_tokenizers_module_imports_without_database_group_deps` case (masking the `database` group, not `nlp`) alongside the existing test.

### [High] `pyutilz.dev.logginglib` — aliased at top level as the generic name `pyutilz.logging` — unconditionally requires the `[database]` stack, undeclared by the `dev` extras group — src/pyutilz/dev/logginglib.py:31-32
- **Category**: architecture / packaging boundary
- **Problem**: `src/pyutilz/dev/logginglib.py:31-32` does `from pyutilz.database.db import safe_execute_values` and `from pyutilz.database.db.sql_helpers import validate_sql_qualified_identifier` at module top level, unguarded. `pyutilz/__init__.py`'s `_MODULE_ALIASES` (lines 36-37) registers this exact module under BOTH `pyutilz.logginglib` and the maximally generic `pyutilz.logging` — a name that invites use by anyone wanting basic Python logging setup, with zero DB connotation. The `dev` extras group (`pyproject.toml:169-192`) declares only pytest/black/ruff/mypy/bandit/py-ci-shared — no sqlalchemy/psycopg2.
- **Failure scenario**: A downstream project does `from pyutilz import logging as pyutilz_logging; pyutilz_logging.init_logging()` (the lazy-alias mechanism documented as the whole point of `_MODULE_ALIASES`) without ever having installed `[database]` → `ModuleNotFoundError: No module named 'sqlalchemy'` on a call that has nothing to do with databases. Verified directly: masking `sqlalchemy`/`psycopg2`/`pymysql` and importing `pyutilz.dev.logginglib` fails immediately with `ImportError: (masked) sqlalchemy`.
- **Suggested fix**: Defer the `pyutilz.database.db` import to the specific function(s) that actually call `safe_execute_values`/`validate_sql_qualified_identifier` (presumably a Redis/Postgres log handler), guarded the same way `spacy`/`nltk` are guarded in `tokenizers.py`.

### [Medium] Three internal source files import via the "backward-compat only" alias `pyutilz.pythonlib` instead of the canonical `pyutilz.core.pythonlib` — src/pyutilz/core/filemaker.py:27, src/pyutilz/data/pandaslib/_common.py:38, src/pyutilz/data/polarslib.py:29
- **Category**: architecture / maintainability
- **Problem**: `pyutilz/__init__.py`'s own header comment (lines 11-12) frames `_MODULE_ALIASES` as existing purely "for backward compatibility" for external/legacy callers of the old flat module layout. Three files inside the library's own source tree use that same alias internally: `from pyutilz.pythonlib import get_attr` / `to_float` / `is_cuda_available, check_cpu_flag` — rather than the canonical `from pyutilz.core.pythonlib import ...` used everywhere else (11 other call sites use the canonical path). This makes the library internally dependent on its own deprecation shim, so the alias can never actually be removed/deprecated without also fixing these three internal call sites, and it's inconsistent style within the same package (`data/polarslib.py` uses the alias for `pyutilz.pythonlib` while separately using the canonical `pyutilz.system.system` on the very next line).
- **Failure scenario**: If a future contributor, following the "backward-compat only, safe to eventually drop" framing in the header comment, deletes the `pythonlib` entry from `_MODULE_ALIASES` (or the alias's lazy proxy silently breaks per the `tokenizers`-shadowing regression class the meta-test already guards against), `pyutilz.core.filemaker`, `pyutilz.data.pandaslib._common`, and `pyutilz.data.polarslib` — none of which are deprecated — would fail with `ImportError: cannot import name 'get_attr' from 'pyutilz.pythonlib'` even though nothing about their own functionality changed.
- **Suggested fix**: Point all three at `pyutilz.core.pythonlib` directly; add a lint rule (ruff `TID` or a small `dev/code_audit` scanner, given the project already has an in-house AST-scanner framework) forbidding internal `src/pyutilz/**` imports from `_MODULE_ALIASES` keys.

### [Medium] pandaslib/polarslib name + contract drift on the same conceptual operation is a silent-no-op porting hazard — src/pyutilz/data/pandaslib/frames.py:378, src/pyutilz/data/polarslib.py:945
- **Category**: architecture / naming consistency
- **Problem**: `pandaslib.frames.remove_constant_columns(df, verbose=False, prewarm_size=10_000) -> None` mutates `df` **in place** (`df.drop(columns=..., inplace=True)` or `del df[var]`, per its own docstring "Drop constant ... columns from `df` in place") and returns `None`. The polars equivalent, `polarslib.drop_constant_columns(df, max_log_text_width=300, verbose=1) -> pl.DataFrame`, does the conceptually identical thing but — because polars frames are immutable — returns a **new** DataFrame and does not touch the caller's `df` at all. Beyond the verb drift (`remove_` vs `drop_`), pandaslib also names its detection helper `get_suspiciously_constant_columns` while polarslib's equivalent-style helpers are named `find_nan_cols`/`find_infinite_cols` (`get_` vs `find_` for the same class of "detect columns with property X" operation), and the dtype-downcast pair `convert_float64_to_float32` (pandaslib) vs `cast_f64_to_f32` (polarslib) drifts both verb ("convert" vs "cast") and abbreviation style.
- **Failure scenario**: A developer who is used to the pandaslib idiom `remove_constant_columns(df)` (call and discard the return value, relying on in-place mutation) ports the same call site to polars as `drop_constant_columns(df)` without capturing the return value (`df = drop_constant_columns(df)`). Because polars frames are immutable, `df` in the caller's scope is completely unchanged — the constant columns silently remain in the pipeline, with no error, warning, or type-checker signal (both functions are valid to call and discard the return of, from mypy's perspective).
- **Suggested fix**: Rename `polarslib.drop_constant_columns` → `remove_constant_columns` (or vice versa) so the two mirror each other exactly, and consider a docstring convention note ("unlike the pandaslib version, this returns a new frame — you must capture it") on every polarslib function whose pandaslib counterpart mutates in place, to make the contract difference explicit at the call site rather than only in a full docstring read.

### [Medium] Three genuinely different, unrelated registry/plugin patterns with no shared abstraction or consistent encapsulation — src/pyutilz/llm/factory.py:30-50, src/pyutilz/dev/code_audit/registry.py:25-40, src/pyutilz/performance/kernel_tuning/registry.py:33,161-193
- **Category**: architecture / consistency
- **Problem**: `llm/factory.py` uses a private, hand-maintained `_PROVIDER_MODULES: dict[name, (module_path, class_name)]` + a *separate* `_ALIASES: dict[alias, canonical]`, resolved lazily via `importlib.import_module` inside `get_llm_provider()` — closed set, no public registration API, no accessor function. `dev/code_audit/registry.py` uses a public, directly-mutable `SCANNERS: dict[str, Callable]` populated by eagerly importing every scanner module at file top — closed set, in-tree only, but exposed as a raw mutable module-level dict with zero encapsulation (any importer can `SCANNERS['x'] = evil_fn` with no guard). `performance/kernel_tuning/registry.py` uses a private `_REGISTRY: dict` populated by *external* callers invoking the public `kernel_tuner(**kwargs)` function at their own module's top level (a genuine open/pluggable registration model, mirrored by mlframe), with a copy-returning `get_registry()` accessor that protects the live dict from external mutation. Each is defensible in isolation given real differences in the domains (closed heavy-SDK set vs. closed lightweight-scanner set vs. open cross-package plugin set) — but only `kernel_tuning` supports external registration, and only it protects its registry from accidental external mutation; the other two would silently break if reused as a template for a hypothetical fourth registry without noticing those two properties don't transfer.
- **Failure scenario**: A future contributor extends `dev/code_audit/registry.py` (the only one of the three with no accessor function) by importing `SCANNERS` directly in a test or a downstream tool and mutating it (`from pyutilz.dev.code_audit.registry import SCANNERS; SCANNERS.pop(...)`) — since nothing enforces a copy, this corrupts the shared dict for every subsequent `run_all()` call in the same process, a class of bug the `kernel_tuning` registry's `get_registry() -> dict(_REGISTRY)` pattern already specifically prevents.
- **Suggested fix**: Not urgent enough to warrant a shared base class given only 3 instances exist, but at minimum give `dev/code_audit/registry.py` a `get_scanners() -> dict` copy-returning accessor matching `kernel_tuning`'s pattern, and rename its module-level dict to `_SCANNERS` (private) to match the other two's naming convention.

### [Medium] No typed-exception convention outside `llm`; `database`/`web`/`system` raise bare `ValueError`/`RuntimeError` for every domain-specific error condition — src/pyutilz/database/redislib.py:54, src/pyutilz/web/proxy/decodo.py:259,280, src/pyutilz/system/gpu_dispatch.py:312
- **Category**: architecture / consistency
- **Problem**: `llm/exceptions.py` defines a real hierarchy (`LLMProviderError` base, `LLMRefusalError`/`LLMSafetyBlockError`/`LLMTruncationError` subclasses) specifically so callers can discriminate retryable vs. non-retryable failures via `except` clauses. No other subpackage has an equivalent module. Grepping every `raise ValueError(...)`/`raise RuntimeError(...)`/`raise Exception(...)` in `database`, `web`, `system` (21 call sites) shows every single domain-specific error condition — "Redis connection is not established" (`redislib.py:54`), "DECODO_API_KEY not set" vs. "Could not fetch subscriptions from any Decodo API endpoint" (`decodo.py:259` vs `280`, both bare `RuntimeError`), "cuda_memory_guard called but no CUDA device is available" (`gpu_dispatch.py:312`) — is raised as a generic built-in with no distinguishing type.
- **Failure scenario**: A caller wrapping `rconnect()`/whatever calls into `redislib` wants to catch "connection not yet established, retry after connecting" specifically and let genuine connection-level Redis errors (`redis.exceptions.ConnectionError`, already imported at `redislib.py:22` but never wrapped) propagate differently. With only a bare `RuntimeError`, the only way to discriminate is `except RuntimeError as e: if "not established" in str(e): ...` — a string-match that breaks silently the moment the message wording changes (no test enforces message stability, unlike the typed-exception case where `except SpecificError` survives a message rewording).
- **Suggested fix**: Not proposing a full hierarchy for every domain (that's real design work, out of scope to mandate wholesale) — but for the two or three highest-traffic domains (`database`, `web`) a small `exceptions.py` mirroring `llm`'s pattern (e.g. `DatabaseConnectionError(RuntimeError)`, `SQLValidationError(ValueError)`) would let callers discriminate without string matching, consistent with the convention the project has already proven out once.

### [Low] `pyproject.toml`'s own documentation comment for the `database` extras group cites a stale file path — pyproject.toml:65-66 vs src/pyutilz/database/db/__init__.py:35
- **Category**: docs / packaging
- **Problem**: `pyproject.toml:65-66` says *"database/__init__.py imports dateutil.relativedelta unconditionally at module level"* — but `src/pyutilz/database/__init__.py` (verified in full: 4 lines, just `__all__ = ["db", "redislib", "deltalakes"]`) has zero imports. The actual unconditional `from dateutil.relativedelta import relativedelta` is one directory deeper, at `src/pyutilz/database/db/__init__.py:35` (alongside unconditional `sqlalchemy`/`psycopg2` imports at lines 32/37-42). This is exactly the class of "now-stale exception comment" the task brief asked to check for — it reads as if it's describing the package's top-level `__init__.py`, but that file was likely a flat `database/db.py` module before being split into the `database/db/` subpackage (matching the project's documented "monolith split via re-export" convention), and the comment's path wasn't updated after the split.
- **Failure scenario**: A future maintainer trusts the comment, edits `database/__init__.py` looking for the `dateutil` import to remove/guard it, finds nothing there, and either gives up or (worse) concludes the documented exception is now obsolete and removes `python-dateutil` from the `database` extras group — which would break `database/db/__init__.py:35` and every caller of `create_postgres_range_partitions`/`delete_postgres_range_partitions` (both use `relativedelta`).
- **Suggested fix**: Change the comment to reference `database/db/__init__.py` (the actual file).

### [Low] TESTING.md and the meta-test's own docstring both claim "24" backward-compat aliases; the actual map has 27 entries — TESTING.md:29, src/pyutilz/__init__.py:13-41
- **Category**: docs / maintainability
- **Problem**: `TESTING.md:29` states flatly "The 24-entry backward-compat module alias map"; `tests/test_meta/test_module_alias_integrity.py:3` hedges with "24+". Counting `_MODULE_ALIASES` in `src/pyutilz/__init__.py:13-41` directly (and confirmed via `len(pyutilz._MODULE_ALIASES)` at runtime) gives **27**. All 27 do resolve to real, importable pyutilz modules (verified: every RHS path corresponds to either an existing `.py` file or an existing subpackage directory with `__init__.py` — `pandaslib`, `db`, `strings` are now subpackages rather than flat modules, which still resolves correctly since Python auto-imports package submodules on `from pyutilz.database import db`-style access) — so this is a stale count, not a functional break. No deprecation timeline exists anywhere for any of the 27 entries; per the header comment they're meant to live indefinitely as a compatibility shim, which is a reasonable design choice for a library whose own downstream project (mlframe, per multiple in-code comments) still depends on the flat import paths, but is worth the maintainer explicitly deciding is permanent rather than "eventually clean up."
- **Failure scenario**: Low-stakes — a contributor reads "24-entry" in TESTING.md, greps `_MODULE_ALIASES` expecting to reconcile a count for a docs PR, finds 27, and either wastes time hunting for a discrepancy or (worse) "fixes" the count down by deleting 3 real, live aliases to make the number match the doc.
- **Suggested fix**: Update `TESTING.md:29` to "27-entry" (or drop the exact number and match the test docstring's "24+" style, though an exact count is more useful and just needs a one-line update whenever an alias is added).

## Things done well

- The lazy PEP 562 proxy-module mechanism in `pyutilz/__init__.py` (`_create_lazy_module`) is genuinely well engineered: the design deliberately avoids writing to the bare leaf key in `sys.modules` specifically to prevent shadowing real third-party top-level packages (the `tokenizers`/HuggingFace regression is documented inline with a precise failure chain), and `test_lazy_proxy_does_not_shadow_real_toplevel_packages` in the meta-suite actively re-tests that exact tripwire.
- `pyutilz/system/__init__.py`'s handling of `distributed` (PEP 562 lazy `__getattr__`, with a very specific inline post-mortem of the mlframe CI breakage it caused before the fix) shows the project genuinely learns from and documents real incidents rather than silently patching them — I'd have liked to see the same discipline applied to `parallel`/`monitoring`/`hardware_monitor` in the same file (see Finding 3), but the `distributed` fix itself is a model example of the pattern done right.
- `llm/exceptions.py`'s typed hierarchy (`LLMProviderError` → `LLMRefusalError` → `LLMSafetyBlockError`, plus standalone `JSONParsingError`/`LLMTruncationError` subclassing `ValueError` so generic `except ValueError` callers still catch them) is a clean, well-thought-out design that correctly encodes retry semantics in the type system rather than in message text.
- `database/db/__init__.py`'s consistent use of `validate_sql_identifier`/`validate_sql_qualified_identifier` before every f-string-interpolated table/column name, with explicit `# nosec B608` annotations explaining exactly why each raw-SQL construction is safe, is careful, systematic defensive coding rather than ad hoc patchwork.
- The top-level `pyutilz/__init__.py` public surface is genuinely curated: `__all__` exposes exactly the 9 subpackages + `__version__`, nothing private leaks through the lazy `__getattr__` (it raises `AttributeError` for anything not in the explicit alias table or subpackage list), and `tests/test_meta/test_api_stability.py` snapshot-guards it.
- `performance/kernel_tuning/registry.py`'s `TunerSpec` dataclass has unusually thorough per-field docstrings and a well-reasoned idempotent-re-registration policy (`kernel_tuner()` overwrites on re-registration with an explicit comment on *why* raising would be wrong, given Python's module-cache semantics under test re-imports).

## Investigated, not an issue

- Suspected `pyutilz/__init__.py`'s `__getattr__`/`_MODULE_ALIASES` mechanism might leak private/internal names alongside the public subpackages — checked the full 81-line file; the alias dict and `__all__` are both explicit, closed, hand-maintained lists, and anything not in either raises `AttributeError`. No leakage.
- Suspected the three registry patterns (`llm/factory.py`, `dev/code_audit/registry.py`, `performance/kernel_tuning/registry.py`) were simply careless copy-paste drift with no underlying rationale — on reading each in full, the differences track real differences in domain shape (closed heavy-SDK set needing lazy import vs. closed pure-stdlib-AST scanner set where eager import is genuinely harmless vs. an open plugin set that external packages populate at their own import time). Still flagged the encapsulation gap as Medium (see Finding 8), but it's not "3 patterns for no reason."
- Suspected `dev/code_audit/registry.py`'s eager import of all 14 scanner modules at package-import time might itself be a heavy/undocumented dependency (mirroring the pattern found elsewhere). Checked `_base.py` and confirmed the scanner framework is pure stdlib (`ast`, `dataclasses`, `pathlib`, `typing`) — eager-importing it costs nothing extra, so this is fine as-is.
- Checked whether `pyutilz.stats/__init__.py`'s eager `from . import normality` (which needs numpy) is an undocumented-dependency violation of the same class as the others above — numpy is a much lighter, more universally-expected dependency than numba/psutil/sqlalchemy, no extras group or README section makes a "stats has zero deps" claim analogous to the "core" one, and this falls closer to the packaging/dependency-management agent's territory than architecture, so I did not write it up as a full finding.
